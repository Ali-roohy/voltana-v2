## Problem

There is no way to verify that OTP delivery (Bale, Telegram, Email) is working without going
through the full user login flow. Debugging a broken Bale bot token or a silent SMTP failure
currently requires log spelunking. Admins need a direct fire-and-confirm test surface.

---

## Goal

Add an admin-only **"تست OTP"** tab to the Settings page that lets the admin send a live test
OTP to their own linked accounts and see success/failure inline — without touching the
production login flow.

---

## Scope In

### 1 — Backend: `POST /v1/admin/test-otp`

New endpoint behind the existing `AdminOnly` middleware.

**Route**: `POST /v1/admin/test-otp`
**Auth**: Bearer JWT required + `AdminOnly` middleware (fresh DB `IsAdmin` check, same pattern as all `/v1/admin/*` routes).

**Request body**:
```json
{ "platform": "bale" | "telegram" | "email" }
```

**Behavior**:
1. Load the authenticated admin's user record (`FindByID`).
2. Resolve the delivery target:
   - `bale` → `users.bale_chat_id` (must be non-null)
   - `telegram` → `users.telegram_chat_id` (must be non-null)
   - `email` → `users.email` (must be non-null)
3. Call the appropriate `OTPSender` directly with a fixed test code (`"000000"`) and a descriptive message body: `«کد تست ولتانا: 000000»`.
   - For `email`: use the `Mailer` interface (`SendOTPEmail`) with a test subject `«تست OTP ولتانا»`.
4. Do **not** write anything to Redis — this is a fire-and-forget test; no real session is created.
5. Return:
   - `200 { "success": true, "message": "OTP sent via bale" }`
   - `400 { "success": false, "message": "bale not linked" }` when the required chat_id/email field is null
   - `500 { "success": false, "message": "<underlying error text>" }` when the sender returns an error

**No new migration.** Uses existing `users` columns (`bale_chat_id`, `telegram_chat_id`, `email`) and existing `OTPSender`/`Mailer` interfaces.

Handler location: `internal/handler/admin_handler.go` (extend existing file if present, or create).
Service method: `AdminService.TestOTPDelivery(ctx, userID, platform)` in `internal/service/admin_service.go`.

---

### 2 — Frontend: Admin tab in Settings

**Visibility**: rendered only when `useMe().data?.is_admin === true`. Non-admin users see no tab, no component, no route — the tab simply does not appear in the Tabs list.

**Tab label**: `تست OTP` (or a wrench icon + label).

**Location**: new file `voltana-web/src/features/settings/AdminOTPPanel.tsx`, imported in the existing `Settings` page (`src/pages/Settings.tsx` or equivalent) and added as a conditional tab.

#### AdminOTPPanel layout

Three cards, one per platform, displayed vertically (full-width on mobile):

```
┌──────────────────────────────────────────────────┐
│  🟣 بله                                          │
│  شناسه: linked / not linked                      │
│  [ ارسال تست ]   ← button, disabled if not linked│
│  ✅ کد ارسال شد  ← shown after success           │
│  ❌ خطا: <message> ← shown after failure         │
└──────────────────────────────────────────────────┘
┌──────────────────────────────────────────────────┐
│  ✈️ تلگرام                                       │
│  (same structure)                                │
└──────────────────────────────────────────────────┘
┌──────────────────────────────────────────────────┐
│  ✉️ ایمیل                                        │
│  آدرس: <email> / not set                         │
│  [ ارسال تست ]                                   │
│  ✅ / ❌                                         │
└──────────────────────────────────────────────────┘
```

**Per-card state** (local `useState`, not persisted):
- `idle` → button enabled/disabled based on link status
- `loading` → button spinner, disabled
- `success` → green badge «✅ کد ارسال شد»
- `error` → red badge «❌ خطا: <message from API>»

State resets to `idle` when the button is clicked again (allows re-test without page reload).

**Data source**: `useMe()` (already in `features/auth/hooks.ts`) provides `bale_linked`, `telegram_linked`, `email`. No new query needed; the panel is read-only re: the user record.

**API call**: `POST /v1/admin/test-otp` via a new function `testOTPDelivery(platform)` in `features/settings/api.ts`. Use the existing `api` helper from `lib/api.ts` (JWT interceptor + silent refresh).

**No TanStack Query mutation** needed — this is a fire-once imperative action. A plain `async` handler in the component is sufficient.

---

## Scope Out

- **Enable/disable toggles per method** — out of scope for this task. Simple fire-and-confirm only; operational toggles require a separate settings field and persistence strategy.
- **Sending a real, session-backed OTP** — the test code `"000000"` is never written to Redis; the admin cannot use it to log in.
- **Non-admin access** — absolutely out. `AdminOnly` on the backend; conditional render on the frontend.
- **Production gating** — this ships to production behind the admin guard; it is "production-safe" because it only fires to the admin's own linked accounts.
- **SMS OTP test** — no SMS sender exists yet; add when SMS is implemented.

- [ ] Settings page shows the **"تست OTP"** tab only when `is_admin === true` (verified by `useMe`).
- [ ] Non-admin users: tab is absent from the DOM entirely.
- [ ] Each card button is disabled when the corresponding channel is not linked/set.
- [ ] Clicking a card button shows a spinner during the request.
- [ ] Success response → green «✅ کد ارسال شد» badge on that card.
- [ ] Error response → red «❌ خطا: <message>» badge on that card.
- [ ] Clicking the button again resets the badge and fires a new request.
