**Created**: 2026-06-12 (pm)

## Problem

The catalog (TASK-0033) is browse-only. A user who finds their actual car in it still has to
create a "My Cars" entry by hand through the slim `ev_models` autocomplete, re-typing what the
catalog already knows — and the two entries stay unrelated. There is no path from "this is my
car" → charging-session tracking.

## Goal

One tap from the catalog detail drawer creates a personalized entry in My Cars: specs pre-filled
from the catalog, user-customizable (their color, their battery variant, …), immediately usable
for charging sessions **and battery analytics**.

---

## Current State — Constraints the Design Must Respect

| Fact | Source | Consequence |
|---|---|---|
| `cars` is slim: `id, user_id, ev_model_id, name, license_plate, odometer_km` | `000001_init_schema` | The 43 catalog specs canNOT become 43 `cars` columns — store a **link + JSONB overrides** |
| SOH analytics resolves nominal capacity ONLY via `car.EVModelID → ev_models` | `analytics_service.go:311 nominalCapacity` (returns false when `EVModelID == nil`) | A catalog-linked car without an ev_model gets **no SOH** unless the resolution chain is extended — **in scope, B3** |
| Chemistry recommendations also read `ev_models.chemistry` | `analytics_service.go` (`recommendations`) | Same fallback applies (`cell_type` from catalog) |
| `PUT /v1/cars/:id` is full-replace | TASK-0004 / TASK-0018 round-trip lesson | New fields must round-trip through the existing edit flow or be silently wiped |
| user_id isolation lives in the repository layer | ADR / TASK-0003 | Unchanged; `catalog_car_id` is a global FK, not a sharing mechanism |

## Scope In

### B1 — Migration `000017_cars_catalog_link`

```sql
ALTER TABLE cars
    ADD COLUMN catalog_car_id UUID REFERENCES ev_catalog(id) ON DELETE SET NULL,
    ADD COLUMN spec_overrides JSONB NOT NULL DEFAULT '{}'::jsonb;
```

- `spec_overrides` stores **only the diff** from the linked catalog car (e.g.
  `{"exterior_color":"آبی متالیک","battery_capacity_kwh":60}`). Effective specs =
  catalog values overlaid with overrides — merged **client-side** (the catalog is already
  client-cached by `useCatalog`); the API does not return a merged view.
- `exterior_color` / `interior_color` are override keys (single chosen value), not new columns.
- Down migration drops both columns. Reverses clean.

### B2 — Extend `POST /v1/cars` + `PUT /v1/cars/:id` (backward-compatible)

Request gains two optional fields:

```json
{
  "name": "بی‌زد4ایکس من",            // optional when catalog_car_id present → defaults to catalog name_fa
  "license_plate": "12ب345-67",
  "odometer_km": 15000,
  "ev_model_id": null,                 // legacy path unchanged; may coexist with catalog link
  "catalog_car_id": "<uuid>",          // optional — must exist in ev_catalog, else 422 INVALID_CATALOG_CAR
  "spec_overrides": {                  // optional — keys whitelisted to the 43 catalog spec keys
    "exterior_color": "آبی متالیک",   //   + exterior_color / interior_color
    "interior_color": "مشکی",
    "battery_capacity_kwh": 60
  }
}
```

- **Validation**: unknown override keys → 400 `INVALID_OVERRIDE_KEY`; values type-checked
  against the catalog field's type (number vs string); numeric overrides that have DB CHECKs in
  `ev_catalog` (capacity > 0, range > 0) enforce the same bounds in the service.
- `Car` domain/response gains `catalog_car_id` + `spec_overrides` (echoed verbatim on
  GET/POST/PUT). **PUT is full-replace** — the frontend edit dialog must round-trip both fields
  (the TASK-0018 odometer lesson).
- New small repo method `CatalogRepository.GetByID(ctx, id)` (404 → 422 mapping in service).

### B3 — Analytics Capacity/Chemistry Fallback

Extend the resolution chain (analytics_service.go) so catalog-added cars get SOH + advice:

1. `spec_overrides.battery_capacity_kwh` (when > 0)
2. linked catalog car's `battery_capacity_kwh` (via `catalog_car_id`)
3. existing `ev_models` path (unchanged)

Same order for chemistry: `spec_overrides.cell_type` → catalog `cell_type` → `ev_models.chemistry`.
Unit tests for the new branches (capacity from override / from catalog / still-none → insufficient).

### F1 — "اضافه کردن به خودروهای من" in the Catalog Detail Drawer

- Button in `CarDetail.tsx` (visible on every tab, e.g. pinned under the banner).
- Opens `features/catalog/components/CustomizeCarModal.tsx`:
  - **My-car fields first**: name (pre-filled `name_fa`), license plate (optional), current
    odometer (km).
  - **Color pickers**: exterior (reuse `ColorPicker`, constrained to that car's
    `exterior_colors`) and interior (`interior_colors`).
  - **All 43 spec fields editable**, pre-filled from the catalog, organized in the same 6
    sections as the Specs tab (General/Battery/Motor/Charging/ADAS/Comfort). Numeric fields as
    number inputs, the rest text. Only **changed** fields are sent as `spec_overrides`.
  - Save → `useCreateCar` (extended `CarInput` in `features/cars/api.ts` — ADR-002, no fetch in
    components) → on success: toast **«خودرو اضافه شد»** + navigate `/cars`.
- `/cars` page renders catalog-linked cars correctly (no `ev_model_id`); show the catalog origin
  (small badge or the catalog name) where the ev-model name appears today.
- The Cars edit dialog must round-trip `catalog_car_id` + `spec_overrides` unchanged (PUT
  full-replace constraint).

### F2 — Charging Works End-to-End

No charging-code changes expected: the session form lists cars from `/v1/cars`. Acceptance just
proves the chain (create catalog car → log session → dashboard/SOH unaffected or improved by B3).
