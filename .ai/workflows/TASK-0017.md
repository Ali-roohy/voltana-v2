# TASK-0017 — OTP Login via Bale/Telegram Bot

**Status**: BACKLOG
**Phase**: 3 (auth feature)
**Assigned to**: architect (design the bot integration first) → feature + developer → dev_supervisor + security
**Reviewer**: dev_supervisor + security (auth boundary, OTP handling, rate limiting, secrets)
**Persona**: pm (this spec) → **architect (bot integration contract — REQUIRED before READY)** → feature (login UI) → developer (implements) → dev_supervisor + security
**Depends on**: TASK-0002 (✅ CLOSED — Go Auth API: JWT issue/refresh, rate limiting, Redis)

## Goal
Add a **passwordless OTP login** path alongside the existing email/password auth: a user enters their phone
number, the Go API sends a **6-digit OTP** to them through a **Bale** (preferred, Iranian) or **Telegram**
(fallback) bot, and they exchange the code for an access token. **Both auth methods stay** — this is additive,
not a replacement.

> **Status note:** held at **BACKLOG** until the **architect** produces the bot-integration contract (see
> "Architect must decide" below). Bot messaging, OTP storage, and the new auth surface have enough security
> and design surface that this should not go to a developer un-designed.

---

## Scope In
- User enters a **phone number** on the login page.
- Go API generates and sends a **6-digit OTP** via the **Bale or Telegram bot** to the user's chat.
- User enters the OTP in Voltana → receives a normal **access token** (same JWT/refresh contract as TASK-0002).
- **OTP expires after 5 minutes.**
- **Rate limit: 3 OTP requests per phone per 15 minutes.**
- **OTP stored in Redis** (keyed by phone), **never in Postgres**.

## Scope Out
- **SMS OTP** — separate task.
- **Replacing email auth** — keep both methods; OTP is additive.
- **User registration via OTP** — Phase 4. This task logs in **existing** users who have a known phone +
  linked bot chat; account creation by OTP is out of scope.

## Tech / Integration
- **Bale Bot API:** `https://api.bale.ai` (Iranian, **preferred**).
- **Telegram Bot API:** `https://api.telegram.org` (**fallback**).
- The user **must have started the bot first** (`/start`) so the bot has a `chat_id` to send to. A user with
  no linked chat cannot receive an OTP — define the UX for that (see open questions).
- Store the **`chat_id` mapped to the phone number** in the `users` table (see DB change).
- Bot tokens are **secrets** — `.env` only (git-ignored), never committed; passed through compose like
  `SMTP_*`/`APP_URL` (TASK-0014 pattern).

## DB Change (migration — next number after 000008)
Add to `users`:
- `phone` (nullable, unique when present — for OTP lookup),
- `telegram_chat_id` (nullable),
- `bale_chat_id` (nullable).

Paired `.up.sql` + `.down.sql`. Index `phone` for the OTP-request lookup. (Architect to finalize column
types/constraints + whether phone is normalized to E.164.)

## API (under the existing auth group; NOT behind the JWT middleware)
- **`POST /auth/otp/request`** — body `{ phone }`. Look up the user + their bot chat_id, generate a 6-digit OTP,
  store in Redis (5-min TTL), send via Bale (fallback Telegram). **Anti-enumeration:** always return `202`
  regardless of whether the phone/chat exists (mirror the resend-verification pattern from TASK-0009).
  Rate-limited **3/phone/15m** (+ a per-IP limit to match existing auth limiters).
- **`POST /auth/otp/verify`** — body `{ phone, code }`. Validate against Redis (constant-time compare,
  single-use → delete on success, expire on TTL), enforce a max-attempts guard, then issue the standard
  access token + httpOnly refresh cookie (reuse the TASK-0002 token service). Generic `401` on bad/expired code.

## Frontend
- New **tab on the login page**: **"ورود با بله/تلگرام"** (login via Bale/Telegram).
- **Phone-number input** → "send code" → calls `/auth/otp/request`.
- **OTP input** (6 digits) → "verify" → calls `/auth/otp/verify` → logs in (same in-memory access token +
  silent-refresh flow as the email path, ADR-003).
- All HTTP through `features/auth/api.ts` + `hooks.ts` (ADR-002 — no `fetch()` in components). Show the
  "start the bot first" guidance + a resend/cooldown timer; keep it RTL/Farsi-consistent with the existing UI.

## Acceptance Criteria
- [ ] Existing user with a linked bot chat: request OTP → receives a 6-digit code in Bale (or Telegram
      fallback) → verify → logged in with a working access + refresh token.
- [ ] OTP **expires after 5 minutes** (verify after expiry → `401`); OTP is **single-use** (a used code is
      rejected on reuse).
- [ ] **Rate limit enforced:** the 4th OTP request for the same phone within 15 min is rejected; `/request`
      still returns `202` for unknown phones (no enumeration).
- [ ] OTP lives **only in Redis** (TTL-bound) — never written to Postgres; bot tokens never committed.
- [ ] Email/password login is **unchanged** and still works (additive).
- [ ] Migration applies cleanly up + down; `phone`/`telegram_chat_id`/`bale_chat_id` added to `users`.
- [ ] `cd voltana-api && go build ./... && go vet ./... && go test ./...` and
      `cd voltana-web && npx tsc --noEmit && npm run build` all pass.

## Architect must decide (gates BACKLOG → READY)
1. **Bot send abstraction** — a `service.OTPSender` interface (Bale impl + Telegram fallback + dev/log mock,
   mirroring the TASK-0009 `Mailer` pattern) and the failover policy (when does Telegram kick in?).
2. **Phone ↔ chat_id linking flow** — how a user links their Bale/Telegram chat to their account (the `/start`
   handshake; do we need an inbound bot webhook/long-poll to capture `chat_id`, and is that in *this* task or a
   prerequisite?). This is the biggest open design risk.
3. **Redis key schema + attempt accounting** — OTP key, rate-limit keys, max-verify-attempts lockout.
4. **Phone normalization** (E.164?) + uniqueness/constraint details for the migration.
5. **Token issuance reuse** — confirm `/auth/otp/verify` reuses the exact TASK-0002 token service + refresh
   cookie contract (no parallel auth path).

## Out of Scope
- Inbound bot command handling beyond what's needed to obtain `chat_id` (architect to scope the minimum).
- SMS, OTP-based registration, replacing email auth (see Scope Out).

## Handoff → architect
Produce the bot-integration contract covering the five "Architect must decide" items, define the
handler→service→repo slice + the `OTPSender` interface, the migration shape, and the Redis schema. Flag the
phone↔chat_id linking flow as the key design decision. On finalize, this moves **BACKLOG → READY** for
feature → developer, with **dev_supervisor + security** as reviewers (security owns the OTP/rate-limit/secret
boundary).

token contract and the Redis/rate-limit infra already in place. **Held at BACKLOG** pending the architect's
