## Goal
Add a **passwordless OTP login** path alongside the existing email/password auth: a user enters their phone
number, the Go API sends a **6-digit OTP** to them through a **Bale** (preferred, Iranian) or **Telegram**
(fallback) bot, and they exchange the code for an access token. **Both auth methods stay** — this is additive,
not a replacement.

> **Status note:** held at **BACKLOG** until the **architect** produces the bot-integration contract (see
> "Architect must decide" below). Bot messaging, OTP storage, and the new auth surface have enough security
> and design surface that this should not go to a developer un-designed.

---

## Scope In
- User enters a **phone number** on the login page.
- Go API generates and sends a **6-digit OTP** via the **Bale or Telegram bot** to the user's chat.
- User enters the OTP in Voltana → receives a normal **access token** (same JWT/refresh contract as TASK-0002).
- **OTP expires after 5 minutes.**
- **Rate limit: 3 OTP requests per phone per 15 minutes.**
- **OTP stored in Redis** (keyed by phone), **never in Postgres**.

## Scope Out
- **SMS OTP** — separate task.
- **Replacing email auth** — keep both methods; OTP is additive.
- **User registration via OTP** — Phase 4. This task logs in **existing** users who have a known phone +
  linked bot chat; account creation by OTP is out of scope.

## Tech / Integration
- **Bale Bot API:** `https://api.bale.ai` (Iranian, **preferred**).
- **Telegram Bot API:** `https://api.telegram.org` (**fallback**).
- The user **must have started the bot first** (`/start`) so the bot has a `chat_id` to send to. A user with
  no linked chat cannot receive an OTP — define the UX for that (see open questions).
- Store the **`chat_id` mapped to the phone number** in the `users` table (see DB change).
- Bot tokens are **secrets** — `.env` only (git-ignored), never committed; passed through compose like
  `SMTP_*`/`APP_URL` (TASK-0014 pattern).

## DB Change (migration — next number after 000008)
Add to `users`:
- `phone` (nullable, unique when present — for OTP lookup),
- `telegram_chat_id` (nullable),
- `bale_chat_id` (nullable).

Paired `.up.sql` + `.down.sql`. Index `phone` for the OTP-request lookup. (Architect to finalize column
types/constraints + whether phone is normalized to E.164.)

## API (under the existing auth group; NOT behind the JWT middleware)
- **`POST /auth/otp/request`** — body `{ phone }`. Look up the user + their bot chat_id, generate a 6-digit OTP,
  store in Redis (5-min TTL), send via Bale (fallback Telegram). **Anti-enumeration:** always return `202`
  regardless of whether the phone/chat exists (mirror the resend-verification pattern from TASK-0009).
  Rate-limited **3/phone/15m** (+ a per-IP limit to match existing auth limiters).
- **`POST /auth/otp/verify`** — body `{ phone, code }`. Validate against Redis (constant-time compare,
  single-use → delete on success, expire on TTL), enforce a max-attempts guard, then issue the standard
  access token + httpOnly refresh cookie (reuse the TASK-0002 token service). Generic `401` on bad/expired code.

## Frontend
- New **tab on the login page**: **"ورود با بله/تلگرام"** (login via Bale/Telegram).
- **Phone-number input** → "send code" → calls `/auth/otp/request`.
- **OTP input** (6 digits) → "verify" → calls `/auth/otp/verify` → logs in (same in-memory access token +
  silent-refresh flow as the email path, ADR-003).
- All HTTP through `features/auth/api.ts` + `hooks.ts` (ADR-002 — no `fetch()` in components). Show the
  "start the bot first" guidance + a resend/cooldown timer; keep it RTL/Farsi-consistent with the existing UI.

## Acceptance Criteria
- [ ] Existing user with a linked bot chat: request OTP → receives a 6-digit code in Bale (or Telegram
      fallback) → verify → logged in with a working access + refresh token.
- [ ] OTP **expires after 5 minutes** (verify after expiry → `401`); OTP is **single-use** (a used code is
      rejected on reuse).
- [ ] **Rate limit enforced:** the 4th OTP request for the same phone within 15 min is rejected; `/request`
      still returns `202` for unknown phones (no enumeration).
- [ ] OTP lives **only in Redis** (TTL-bound) — never written to Postgres; bot tokens never committed.
- [ ] Email/password login is **unchanged** and still works (additive).
- [ ] Migration applies cleanly up + down; `phone`/`telegram_chat_id`/`bale_chat_id` added to `users`.
- [ ] `cd voltana-api && go build ./... && go vet ./... && go test ./...` and
      `cd voltana-web && npx tsc --noEmit && npm run build` all pass.

# Architect Contract — FINALIZED (2026-06-03)

Grounded against the live codebase: migrations in `./migrations/` (next = **000010** — `000009` was taken by
TASK-0018's `charging_sessions.odometer_km`); `RedisTokenStore`
already provides `CacheGet/CacheSet/CacheDel`, `GetDel` (single-use), and `CheckRateLimit(key,max,window)`
(atomic fixed-window); `Mailer` is an interface wired by env in `main.go` (`SMTP_HOST` set → real, else
`LogMailer`); token issuance lives inside `AuthService.Login`. The OTP feature mirrors these patterns — no
new infra.

## DECISION 1 — Phone ↔ chat_id linking flow  ⭐ (the key risk)

**Linking is an authenticated, bot-verified handshake — not a typed phone number.** A user cannot type a
phone for OTP login until they have first *linked* a bot chat while logged in via email/password. Flow:

1. **Initiate (authenticated):** logged-in user opens **Settings → "اتصال بله/تلگرام"**. Frontend calls
   **`POST /v1/account/bot-link`** (JWT-protected). Server mints a random **link token**, stores
   `botlink:<token> → userID` in Redis (TTL 10 min), and returns a **deep link** per platform:
   - Bale: `https://ble.ir/<BALE_BOT_USERNAME>?start=<token>`
   - Telegram: `https://t.me/<TG_BOT_USERNAME>?start=<token>`
2. **Open in the messenger:** the user taps the link → the messenger opens the bot with `/start <token>`.
3. **Bot captures chat_id + phone:** the bot backend (long-poller, Decision 3) receives the `/start <token>`
   update. It resolves `botlink:<token>` → userID (`GetDel`, single-use). It then replies with a
   **"اشتراک‌گذاری شماره تلفن" (share contact)** button (`reply_markup` with `request_contact`). When the user
   taps it, the messenger sends a **contact** update whose `phone_number` is **verified by the platform**.
4. **Persist:** the bot backend writes to that user row: the platform `chat_id`
   (`bale_chat_id`/`telegram_chat_id`) **and** `phone` (normalized to E.164). The bot replies "اتصال انجام شد".

**Why this design:** the phone is taken from the platform's **verified contact**, never typed during linking —
so the phone↔account binding is trustworthy and can't be spoofed. The link token ties the chat to an
**already-authenticated** user, so there's no hijack/enumeration window. This **also** doubles as phone capture,
removing a separate phone-verification step. (Typing a phone during linking would require verifying ownership =
OTP = circular.)

**Scope correction (flag to PM):** OTP login is unusable without this linking mechanism, which did not exist.
The minimal inbound bot handling (`/start <token>` + contact share) is therefore **IN scope** for TASK-0017.
**Recommend two developer slices** under this one task:
- **Slice A — Linking:** migration 000010 · `POST /v1/account/bot-link` · the long-poller + `/start`/contact
  handler · Settings "link" UI.
- **Slice B — OTP login:** `POST /auth/otp/request` + `/auth/otp/verify` · `OTPSender` · login-page tab.
If PM prefers, Slice A can split out as **TASK-0017a** (linking) with 0017b depending on it. Architect
recommendation: **keep as one task, build A before B.**

## DECISION 2 — `OTPSender`: Bale-first, Telegram fallback (NOT user-chooses-per-login)

The user already "chose" by what they linked. No per-login platform picker (friction). Resolution per send:

- **Send to Bale if `bale_chat_id` is set; else Telegram if `telegram_chat_id` is set.**
- **Failover:** if the Bale send returns a transport/API error **and** a `telegram_chat_id` exists, retry once
  on Telegram. (Preference order *and* resilience.)

Contract (signature-level — developer implements):
```
type Platform string // "bale" | "telegram"
type OTPSender interface {
    Platform() Platform
    Send(ctx context.Context, chatID, code string) error
}
```
- Concrete: `BaleSender` (POST `https://api.bale.ai/bot<token>/sendMessage`), `TelegramSender`
  (`https://api.telegram.org/bot<token>/sendMessage`) — Bale's Bot API mirrors Telegram's.
- A `service`-level resolver picks Bale→Telegram from the user's linked chat_ids and orchestrates failover.
- **Dev `LogOTPSender`** (mirrors `LogMailer`): used when **no bot token is configured**; logs the code to the
  server log **in dev only** so QA can test without a real bot. Real senders **never log the code**.

**Wiring (`main.go`, mirrors the Mailer block):** `BALE_BOT_TOKEN` set → register `BaleSender`;
`TELEGRAM_BOT_TOKEN` set → register `TelegramSender`; neither → `LogOTPSender`. Tokens flow through compose +
`.env` like `SMTP_*` (TASK-0014 pattern); **never committed**.

**Prod reachability flag (security/release):** **Telegram's API is filtered in Iran** — on the Iranian VPS the
Telegram fallback may be unreachable without a proxy. **Bale is the production primary**; Telegram is realistically
a dev/diaspora path. Do not rely on Telegram as the prod failover without a confirmed egress route.

## DECISION 3 — Long polling (NOT webhook) for this host

**Use `getUpdates` long polling**, run as a background goroutine in the api process (started in `main.go`,
only when a real bot token is configured; off under `LogOTPSender`). One poller per configured platform.

**Rationale:** a webhook needs a **public HTTPS URL** the bot servers can reach — the **WSL2 dev host is behind
NAT** (not reachable) and a webhook adds an unauthenticated public endpoint (attack surface). Long polling is
**outbound-only HTTPS**, works identically on the WSL2 dev box and the small VPS, needs no nginx route/TLS for
the bot, and fits the project's in-process worker preference (cf. TASK-0007's synchronous recompute, no asynq).
The poller's **only** responsibilities (minimal): handle `/start <token>` and contact-share updates for linking.
Arbitrary bot commands remain out of scope. Revisit webhooks only if poll latency/load matters at scale.

## DECISION 4 — Migration `000010_users_phone_chat_ids` (.up / .down)

`up`:
- `ALTER TABLE users ADD COLUMN phone TEXT;` — **E.164**, normalized with leading `+` (e.g. `+98912...`).
- `ALTER TABLE users ADD COLUMN bale_chat_id TEXT;`
- `ALTER TABLE users ADD COLUMN telegram_chat_id TEXT;`
- `CREATE UNIQUE INDEX uq_users_phone ON users (phone) WHERE phone IS NOT NULL;` — partial-unique (many NULLs
  allowed, non-null phones unique); also serves the OTP-request lookup.
- `CREATE UNIQUE INDEX uq_users_bale_chat ON users (bale_chat_id) WHERE bale_chat_id IS NOT NULL;`
- `CREATE UNIQUE INDEX uq_users_telegram_chat ON users (telegram_chat_id) WHERE telegram_chat_id IS NOT NULL;`
  (prevents one bot chat binding to multiple accounts).

`down`: drop the three indexes, then the three columns. (Schema v8 → **v9**.)

`chat_id` as **TEXT** (Telegram/Bale ids are large ints — TEXT avoids int64/overflow fuss and compares fine).
**Phone normalization is server-authoritative:** accept Iranian inputs (`0912…`, `98912…`, `+98912…`) and
canonicalize to E.164 on both ingest (contact share) and lookup (OTP request). The client may pre-format but
the server's normalization is the source of truth.

## Redis key schema (single source — all TTL-bound, nothing in Postgres)

| Key | Value | TTL | Notes |
|-----|-------|-----|-------|
| `otp:login:<e164>` | 6-digit code | 300s | single-use → `GetDel` on verify |
| `otp:attempts:<e164>` | int counter | 300s | lock after **5** wrong tries → delete OTP, force re-request |
| `otp:rl:phone:<e164>` | counter | 15m | `CheckRateLimit(max=3)` — the 4th request is rejected |
| `otp:rl:ip:<ip>` | counter | 15m | `CheckRateLimit(max=10)` — matches existing auth IP limiter |
| `otp:cooldown:<e164>` | "1" | 60s | resend cooldown; still return 202 if present |
| `botlink:<token>` | userID | 600s | linking deep-link token; `GetDel` single-use by the poller |

## Token issuance reuse (no parallel auth path)

Extract the token-pair generation currently inside `AuthService.Login` into a shared private helper, e.g.
`issueSession(ctx, user) (access, refresh string, err error)`, and call it from **both** `Login` and a new
`CompleteOTPLogin(ctx, phone, code, ip)`. The OTP verify **handler** sets the refresh cookie via the existing
`setRefreshCookie` and returns `tokenResponse{access_token}` — **identical** to `/auth/login`. One JWT signer,
one refresh-rotation/TTL policy (ADR-003). No new token semantics.

## Endpoint contracts (final)

- **`POST /v1/account/bot-link`** (JWT) → `{ bale_url, telegram_url }` (only the platforms with a configured
  bot token). Mints `botlink:<token>`.
- **`POST /auth/otp/request`** `{ phone }` → **always `202`** (anti-enum). Normalize phone → look up user +
  linked chat → if found, generate+store OTP, set cooldown, send (Bale→Telegram). Rate-limited by phone (3/15m)
  + IP (10/15m). Unknown phone / no chat / cooldown active → still `202`, no send.
- **`POST /auth/otp/verify`** `{ phone, code }` → `200` + access token + refresh cookie on success
  (single-use `GetDel`); generic **`401`** for wrong/expired/missing code; increment `otp:attempts`, lock at 5.
  No distinction between "no OTP" and "wrong code".

## Open risks / prerequisites (flag to operator + security)

1. **Bale Bot API parity** — design assumes Bale mirrors Telegram's Bot API (`getUpdates`, `sendMessage`,
   `request_contact` reply keyboard, `/start` deep-link param). Developer must confirm against Bale's docs
   (`https://docs.bale.ai`) before Slice A; if Bale lacks `request_contact`, fall back to capturing phone a
   different way (flag back to architect).
2. **Bot credentials for QA = BLOCKER for full live smoke.** A real **Bale** (and optionally Telegram) bot
   token + a test account are needed end-to-end. The `LogOTPSender` dev path covers OTP request/verify logic
   without a bot, but the **send + linking handshake** need a real bot. **Operator must provision a Bale bot
   token** (BotFather-equivalent on Bale) or QA accepts a `LogOTPSender`-based caveat for the send leg.
3. **Telegram egress from the Iranian VPS** (Decision 2) — fallback may be unreachable in prod.
