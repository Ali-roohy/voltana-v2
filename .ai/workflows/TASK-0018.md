# TASK-0018 — Odometer Tracking for Efficiency Metrics

**Status**: BACKLOG
**Phase**: 3 (feature)
**Assigned to**: developer
**Reviewer**: dev_supervisor
**Persona**: pm (this spec) → developer (implements) → dev_supervisor → qa → qa_supervisor
**Depends on**: TASK-0004 (✅ charging sessions CRUD) · TASK-0008 (✅ dashboard analytics — the avg-efficiency surface)

## Goal
Let users record an **odometer reading** with each charging session so Voltana can compute real driving
efficiency (**kWh/100km**) — per session and as a fleet average — instead of showing a placeholder. The
`charging_sessions.odometer_km` column already exists; this task wires it through the API, the session form,
the session card, and the dashboard efficiency card.

---

## Scope In
- Add an **optional `odometer_km`** field to the charging-session form.
- Persist to **`charging_sessions.odometer_km`** (column already exists in the DB — verify before adding any
  migration).
- Compute **kWh/100km per session** when an odometer reading is available (and a prior reading exists to
  derive distance).
- Show the per-session efficiency on the **session card**.
- Show a **fleet average efficiency** across all sessions that have the data on the **dashboard**.

## Scope Out
- **OBD auto-fill** of the odometer (Phase 4).
- **Sub-session / GPS per-trip efficiency** (instantaneous trip readings). Per-session efficiency derived from
  two consecutive odometer readings **is in scope** (see the note reconciling this with Acceptance below).

## Acceptance Criteria
- [ ] `odometer_km` is **optional** — existing sessions (and new ones left blank) are unaffected; no
      regression to create/update/list.
- [ ] When two **consecutive** sessions (for the same car, by time) both have an odometer reading, compute
      **`km_driven = odometer_new − odometer_old`**.
- [ ] **`kWh/100km = (session_kwh / km_driven) × 100`** (guard `km_driven > 0`; ignore non-positive/﻿zero).
- [ ] The session card shows **"X kWh/100km"** for sessions where it can be computed.
- [ ] The dashboard **average-efficiency card shows real data** (no longer "—") once odometer data exists.
- [ ] `cd voltana-api && go build ./... && go vet ./... && go test ./...` and
      `cd voltana-web && npx tsc --noEmit && npm run build` all pass.

## Notes / pointers
- **`odometer_km` column already exists in `charging_sessions`** (per the original schema). Developer must
  **confirm on disk** (`\d charging_sessions` / the migration) before assuming — if absent, that's a scope
  change (new migration) to flag back to pm.
- **Reconcile Scope Out vs Acceptance:** "per-trip efficiency" in Scope Out means *sub-session/GPS* readings,
  which are out. The Acceptance formula (distance from two **consecutive session** odometer readings) is the
  in-scope per-session metric — implement that.
- **Dashboard avg-efficiency surface:** TASK-0008 added `avg_kwh_per_100km` (null when `total_km == 0`) and
  TASK-0011 repurposed the dashboard's old avg-efficiency *card* → avg cost/session. Developer to confirm the
  current dashboard wiring and decide whether to (a) revive the efficiency value on an existing card or
  (b) surface it where the analytics API already exposes it — without breaking the 0011 cost card. Keep the
  computation server-side in the analytics service where the totals already live.
- Per-session efficiency attributes a session's added energy to the distance driven since the **previous**
  odometer reading (a standard approximation); compute it where the session list is assembled (service layer),
  not in the component.
