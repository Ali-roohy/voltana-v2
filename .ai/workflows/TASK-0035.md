## Problem

There are now **two parallel "add a car" flows** with very different quality:

1. **`/cars` add dialog** (TASK-0003 era): searches the slim `ev_models` table (3 usable spec
   fields: brand, capacity, range), links `ev_model_id`, defaults the car name to **`name_en`**.
2. **`/catalog` → CustomizeCarModal** (TASK-0034): picks from the verified 23-car `ev_catalog`
   (43 specs), links `catalog_car_id` + overrides, defaults the name to **`name_fa`**.

Users see different car lists, different spec depth, and different naming defaults depending on
which door they enter through. The ev_models↔ev_catalog open question was flagged at both
TASK-0033 and TASK-0034 close; this task resolves it **at the UX level**: the catalog becomes the
single source for *new* car links, while `ev_models` stays as read-only legacy support.

## Goal

One mental model: "pick your car from the catalog." The `/cars` add dialog searches the catalog;
`ev_models` disappears from the add path (but keeps serving existing cars' display and the
analytics fallback).

## Scope In (frontend-only)

### F1 — Catalog picker in the `/cars` add dialog

In `pages/Cars.tsx`, replace the ev_models search block (create mode only) with a catalog picker:

- Search input filters the 23 cached catalog cars client-side on `name_fa` / `name_en` / `brand`.
- Result rows show: `name_fa` (+ `name_en` small), brand · battery kWh · range km · cell type —
  richer than the old 3-field rows, same visual pattern (Card list under the input).
- Selecting a row sets `formData.catalog_car_id` (NOT `ev_model_id`) and defaults the name to
  **`name_fa`** when the name field is blank (align with CustomizeCarModal).
- A small summary line under the picker shows the chosen catalog car (like the current
  "مدل: …" line) with a clear/unlink ✕.
- Manual path stays: a car can still be created with just a name (no link at all).
- **No new ev_model selections** anywhere in the UI after this task.
- Optional hint under the picker: link to `/catalog` "برای شخصی‌سازی کامل از کاتالوگ اضافه کنید"
  (deep customization stays in CustomizeCarModal — the quick-add here sends empty overrides).

### F2 — Legacy cars keep working untouched

- Cards: ev_model-linked cars keep showing the model name (existing `modelById` path);
  catalog-linked cars keep the «کاتالوگ» badge (TASK-0034).
- Edit dialog: keeps round-tripping `ev_model_id` AND `catalog_car_id`+`spec_overrides`
  verbatim (PUT full-replace). Editing a legacy car must not strip its `ev_model_id`.
- `features/ev-models/` and `GET /v1/ev-models*` remain in place (display + analytics fallback);
  mark the feature's `api.ts` with a deprecation comment pointing new code at `features/catalog`.
