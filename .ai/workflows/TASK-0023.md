# TASK-0023 — BUG: Total Distance Not Updating

**Status**: DONE ✅ CLOSED (qa_supervisor 2026-06-03)
**Phase**: 3 (bug fix)
**Assigned to**: developer
**Reviewer**: dev_supervisor
**Persona**: pm (this spec) → developer (fix + regression test) → dev_supervisor → qa
**Depends on**: TASK-0018 (✅ session odometer + `EfficiencyAggregateByUser`)

## Bug Description
The **"Total km" card on the dashboard stays fixed** even after new charging sessions are logged. It
does not update to reflect newly-driven distance. This erodes trust in the dashboard metrics.

---

## Root Cause (confirmed by pm review of `analytics_service.go`)

```go
// analytics_service.go lines 118–121 (current, broken)
totalKM := 0
for _, c := range cars {
    totalKM += c.OdometerKM   // ← reads cars.odometer_km: static, only changes when the user
}                              //   manually edits the car record
```

`totalKM` is computed from **`cars.odometer_km`** — the odometer field on the car entity, which is a
manually-entered static value that never auto-increments from charging sessions. A user who logs 5
new sessions with odometer readings sees no change to the "Total km" card because the car's odometer
field hasn't changed.

TASK-0018 introduced `EfficiencyAggregateByUser` which sums the **session-derived odometer deltas**
(consecutive `odometer_km` pairs). That same `effKM` value — already computed a few lines below —
is the correct source for `totalKM`.

## Fix
In `internal/service/analytics_service.go`, **remove the car-odometer loop** and derive `TotalKM`
from the session-odometer data instead:

```go
// Before:
totalKM := 0
for _, c := range cars {
    totalKM += c.OdometerKM
}
stats := &domain.DashboardStats{..., TotalKM: totalKM, ...}

// After:
stats := &domain.DashboardStats{..., TotalKM: 0, ...}  // default 0
...
effKWh, effKM, err := s.sessions.EfficiencyAggregateByUser(ctx, userID)
...
if effKM > 0 {
    stats.TotalKM = int(math.Round(effKM))   // update from session deltas
    eff := round2(effKWh / (effKM / 100))
    stats.AvgKWhPer100KM = &eff
}
```

Note: this changes the **semantics** of `TotalKM` from "sum of car odometers" to "total km derived
from charging session odometer pairs". The latter is more useful (it reflects actual driven distance
logged in the app) and is already the data source for `AvgKWhPer100KM`.

## Side Effects to Verify
- `domain.DashboardStats.TotalKM` is `int` — `math.Round(effKM)` is safe.
- The `cars` list fetch (`s.cars.ListByUser`) is still needed for the car count implicitly used
  by the SOH recomputer — check if it's used anywhere else in `DashboardAggregates`. If the only
  use was `totalKM`, the loop can be removed but the list fetch may still be needed for car count
  returned by `ListByUser`. **Developer to verify and remove the loop if the list is still needed
  for other reasons, or remove the whole `ListByUser` call if it's now unused.**
- The dashboard Redis cache key is per-user (`analytics:dashboard:<userID>`) and is busted on
  charging session writes (wired via `chargingSvc.SetHealthRecomputer`). No change needed there.

## Regression Test
Add a test to `analytics_service_test.go` (or a new `_test.go` in `internal/service`) that:
1. Creates two sessions for the same car with consecutive odometer readings (e.g., 1000 km, 1100 km).
2. Calls `DashboardAggregates` and asserts `TotalKM == 100` (not 0 or the car's static odometer).
3. Adds a third session (odometer 1200 km) and asserts `TotalKM == 200`.

This verifies the fix and guards against regression when the analytics service is refactored.
