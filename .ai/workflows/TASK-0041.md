## Context

TASK-0040 shipped the VPS configs with **placeholders** for the runtime secrets:
`VAPID_PUBLIC_KEY` / `VAPID_PRIVATE_KEY` (web push) and `BALE_BOT_TOKEN` /
`BALE_BOT_USERNAME` (+ optional Telegram). The dev `.env` already has a working set of all
six; the **VPS `.env` does not**. This task produces the real values the operator pastes
into `/opt/voltana/.env` for go-live, and adds a reproducible VAPID generator so it never
has to be done by hand again.

**Naming note (resolving the title):** "dev VAPID" here means *generated-by-us* keys (vs.
a third-party service). Per TASK-0040, the VPS must get a **fresh** pair — **do NOT reuse
the dev `.env` keys** on the VPS.

**Two different kinds of secret:**
- **VAPID keypair** — we can generate this ourselves (deliverable below).
- **Bot tokens** — these are issued by **BotFather** (Bale primary; Telegram optional) and
  are tied to the operator's bot accounts. They **cannot be generated in-repo** — see the
  operator-dependency blocker.

## Scope In

### Part A — VAPID generator (`cmd/genvapid`) + prod pair
- Add `voltana-api/cmd/genvapid/main.go` — a tiny CLI that calls
  `webpush.GenerateVAPIDKeys()` (the `SherClockHolmes/webpush-go v1.3.0` lib is already in
  `go.mod`) and prints:
  ```
  VAPID_PUBLIC_KEY=...
  VAPID_PRIVATE_KEY=...
  ```
  This makes the command **already promised** in `voltana-api/.env.example`
  (`go run ./cmd/genvapid`) real — right now that path 404s.
- Promote `webpush-go` from `// indirect` to a direct dependency (`go mod tidy` after the
  new import).
- Generate a **fresh production pair** with the helper and hand the values to the operator
  (out-of-band — see Part C). The frontend needs no rebuild: it fetches the public key at
  runtime from `GET /v1/push/vapid-key`.

### Part B — Bot tokens (operator-provisioned)
- **Bale (primary):** operator creates/locates the bot via BotFather on Bale, copies
  `BALE_BOT_TOKEN` + the bot `@username` → `BALE_BOT_USERNAME` (without `@`).
- **Telegram (optional fallback):** same via @BotFather on Telegram. Expected to be filtered
  on an Iranian VPS, so leave commented unless wanted.
- **Token hygiene:** if the dev tokens were ever exposed in pre-TASK-0038 logs (see the
  bot-token log-leak history), **rotate** them in BotFather and use the rotated values on the
  VPS. Do not paste tokens into chat, commits, or the terminal echo.

### Part C — Record on the VPS + verify
- Operator sets all six values in `/opt/voltana/.env` (the TASK-0040 `.env.production.example`
  block documents each). Nothing secret is committed.
- After `bash scripts/deploy.sh`, verify on the VPS:
  - API log shows `push: web push configured (VAPID)` (not the "disabled" line).
  - `curl -s https://voltanaev.ir/v1/push/vapid-key` (with an auth token) returns the
    `vapid_public_key`.
  - Request an OTP via Bale → code is delivered by the bot (Iranian VPS = reachable).
