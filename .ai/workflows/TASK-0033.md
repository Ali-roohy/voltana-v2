## Problem

Voltana has no EV catalog. Users adding a car pick from the bare `ev_models` autocomplete; there is
no way to browse, filter, or compare the EVs actually available on the Iranian market. We have a
verified dataset (`car_verified__4_.xlsx`, 23 EVs × 43 specification columns — Persian/English names,
battery, motor, charging, ADAS, colors, dimensions) that is currently unused.

## Vision

Transform `car_verified__4_.xlsx` into a stunning, interactive EV catalog where users browse and
compare 23 EVs with 43 detailed specifications. The app theme dynamically adapts to the selected
car's exterior body color.

Design inspiration: Tesla Model Viewer (immersive hero, organized specs), Figma car-spec sheets
(color-coded attributes), Apple product pages (clean, generous whitespace), Iranian EV dealership
sites (Persian-first UX, detailed specs).

# Developer Implementation (2026-06-12)

## Backend (B1 + B2)
- **`scripts/seed-ev-catalog.py`** — stdlib-only converter (zipfile + ElementTree; no
  pandas/openpyxl needed — workbook uses inline strings). Maps all 43 columns by index,
  strips units ("266 Nm", "7.5s", "452 L"), treats `نامشخص/—/empty` as NULL, splits
  exterior colors on `،` and interior on `/`, escapes quotes, derives `img_url` slug.
  Regenerates `migrations/000016_ev_catalog.up.sql` in one command.
- **Migration `000016_ev_catalog`** — table (44 data cols incl. derived img_url) +
  `name_en` UNIQUE + brand/body_type indexes + `set_updated_at` trigger + 23 seed rows in
  the same file. Down: `DROP TABLE`.
- **Go slice** — `domain.CatalogCar` (new file `domain/ev_catalog.go`; NOT `domain.Car`,
  which already exists for fleet cars), `repository/catalog_repo.go` (pgxpool, NUMERIC→
  ::float8 casts per evModelCols convention), `service/catalog_service.go` (cache-aside on
  the shared `CacheStore` iface, key `catalog:cars`, TTL 1 h, cache errors non-fatal),
  `handler/catalog_handler.go` (apiError envelope), route `v1.GET("/cars/catalog")` inside
  the authed group. +3 service tests (`catalog_service_test.go`): cache-aside hit, corrupt
  cache recompute, repo error propagation.

## Frontend (F1–F5)
- **`features/catalog/`**: `api.ts` (CatalogCar iface, `getCatalog`), `hooks.ts`
  (`useCatalog`, staleTime 30 m), `format.ts` (fa-IR digits, null→"—", segment tier, sort,
  battery buckets), `CatalogPage.tsx` (route `/catalog`, grid/list toggle persisted to
  `voltana:catalog-view`, brand/body/segment/battery filters + range/accel/tier sort —

  all 43 fields in the 6 spec sections), `ColorPicker.tsx` (framer-motion swatch pills),
  `CarComparison.tsx` (≤3 cars, 15 metric rows, direction-aware best=green/worst=red).
- **`lib/dynamic-theme.ts`** — Persian color name → keyword table → hue → warm/cool/neutral

## Deviations from the operator sketch (all pattern-driven)
1. Migrations live in top-level `migrations/` (not `voltana-api/internal/migrations/`).
2. `domain.CatalogCar`, not a second `domain.Car` (name collision would not compile).
3. Repo takes `*pgxpool.Pool` (house pattern), not `*pgx.Conn`; handler uses the
   `apiError` envelope, not `gin.H{"error": err.Error()}` (no internal-error leakage).
4. Converter is stdlib Python (`scripts/seed-ev-catalog.py`), not pandas — pandas is not
   installed and the inline-string format doesn't need it. Column indices in the sketch
   were off; corrected against the verified header (43 cols, see Preconditions).
5. Route registered **inside** the authed `/v1` group per the spec (sketch had it public).
6. Card button uses «جزئیات» (sketch text was garbled).
7. Nullable specs are pointers/null (spec: "null, never 0").
