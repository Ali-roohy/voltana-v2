# TASK-0021 — Currency Unit in Settings

**Status**: DONE ✅ CLOSED (qa_supervisor 2026-06-04)
**Phase**: 3 (feature)
**Assigned to**: developer
**Reviewer**: dev_supervisor
**Persona**: pm (this spec) → developer (implements) → dev_supervisor
**Depends on**: TASK-0005 (✅ user_settings CRUD API) · TASK-0010 (✅ `lib/cost.ts` + cost display)

## Goal
Let users choose their preferred currency unit (**Toman / Rial / USD**). The selection is saved in
`user_settings` (persisted per-user, synced across devices) and applied to **every cost display** in the
app: dashboard cards, session cards, TOU breakdown, trend charts.

Currently all costs are shown in **Toman** (hard-coded since TASK-0010 with no ÷10). This task makes
the unit configurable without changing any stored values — conversion is display-only.

---

## Scope In

### Backend
- **Migration `000011`**: add `currency TEXT NOT NULL DEFAULT 'toman'` with a `CHECK (currency IN ('toman', 'rial', 'usd'))` constraint to the `user_settings` table. Paired `.up.sql` + `.down.sql`.
- **`domain.UserSettings`**: add `Currency string` field.
- **`settings_repo`**: include `currency` in SELECT and in the full-replace PUT (already a full-replace,
  so it just gains a new column).
- **`GET /v1/settings`** and **`PUT /v1/settings`**: expose `currency` in request/response (the handler
  already passes the whole struct through; adding the field is sufficient).

### Frontend
- **`features/settings/api.ts`**: add `currency` to `Settings` and `SettingsUpdate` interfaces.
- **`lib/cost.ts`**: extend `formatCost(amount, currency)` to accept a currency argument and apply:
  - `'toman'` → display as-is (current behavior): "۱۲۳ تومان"
  - `'rial'` → multiply by 10, append "ریال"
  - `'usd'` → divide by the Toman/USD rate from settings (or a static fallback constant until a live
    rate feed is in scope — **flag to pm**: static vs live rate is a scoping decision).
- **Currency selector** in `pages/Settings.tsx`: three-button toggle (Toman / Rial / USD) in the rates
  card or a new "نمایش" (Display) card. Wired to `useUpdateSettings`.
- Replace every hard-coded "تومان" / "Toman" string in the app with `formatCost(amount, settings.currency)`.
  Key locations: `pages/Index.tsx` (dashboard cards), `pages/Charging.tsx` (session cards + TOUBreakdown),
  `lib/cost.ts` (its own `formatCost` output).
- `useSettings` is already loaded in most pages; pass `settings.currency` to `formatCost`.

## Scope Out
- Live exchange rate feed for USD (use a static rate constant for now; flag separately).
- Custom currency (arbitrary symbol / rate).
- Historical conversion (all stored amounts stay in Toman; conversion is display-only, not stored).

## Open Question (flag to developer)
**USD rate source**: should the USD display use a hardcoded fallback rate (e.g. `1 USD = 500,000 Toman`)
updated by the developer at spec time, or should the user enter their own rate in Settings? Recommend
**user-entered rate** in Settings alongside the currency selector (one extra field: `usd_rate`), so the
value doesn't go stale. If pm prefers a static constant, that's an acceptable simplification — developer
to flag before implementing and wait for operator confirmation.

is the formatCost refactor, which touches many components but is low-risk (existing behavior unchanged
unless the user actively changes the setting). TASK-0010's `lib/cost.ts` is the leverage point — all
cost display already routes through it. No architect pass needed (extends existing settings contract).
