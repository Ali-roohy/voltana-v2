## ARCHITECTURE REQUIREMENT (binding for the whole task)

**`estimateConsumption(context)` — one extensible function.** All consumption estimation MUST flow
through a single function that today returns the **car's average kWh/100km** but accepts a `context`
object so later signals plug in **without a rewrite**:

```ts
// voltana-web/src/features/charging/consumption.ts  (or a lib/ equivalent)
type ConsumptionContext = {
  car: { avgKwhPer100km: number; usableCapacityKwh: number };
  month?: number;          // Jalali month (FEAT-2 season multiplier)
  regenFactor?: number;    // FEAT-4
  sameLocationAsPrev?: boolean; // FEAT-4 elevation heuristic
  // future: elevation delta, temperature, payload …
};
function estimateConsumption(ctx: ConsumptionContext): number; // kWh/100km
```

- FEAT-1 uses only `car.avg`. FEAT-2 multiplies by the seasonal factor. FEAT-4 applies
  `× (1 − regenFactor)`. **No feature may fork its own consumption math** — each adds a term to the
  same function/context. dev_supervisor rejects any slice that duplicates consumption logic.
- **BUG-1 mirror rule for dates:** every date/month rendered anywhere in the app goes through
  `src/lib/dates.ts`. No hardcoded Jalali or Gregorian formatting outside that file. Adding English
  later must flip every date to Gregorian with zero per-component edits.

### BUG-4 — Odometer is cumulative: enforce monotonic + fix distance/efficiency — HIGH
Operator-confirmed: `odometer_km` is a **cumulative** odometer → must strictly increase per car.
- **Create/update validation:** if `odometer_km <= previous session's odometer for the same car` →
  **REJECT** (422) with message **"کیلومترشمار باید از جلسه قبلی بیشتر باشد (کیلومترشمار تجمعی است)"**.
  Enforce in `service/charging_service.go` (needs the previous odometer for the car — add a repo
  lookup or reuse the LAG path). "Previous" = the immediately-earlier session by `started_at` for
  that car; inserting a back-dated session must validate against its true chronological neighbor.
- **Store `trip_distance_km`** (new column, see Migration) = `current_odometer − previous_odometer`
  on write. Backfill existing rows from consecutive odometers.
- **Odometer absent:** derive `trip_distance_km = (soc_delta / 100) × effective_range_km`, where
  `effective_range_km` resolves override → catalog → ev_model (same chain as
  `nominalCapacity`/`chemistryOf` in `analytics_service.go`).
- **Efficiency sanity guard:** when aggregating the fleet `avg_kwh_per_100km`
  (`repository.EfficiencyAggregateByUser`), **ignore any pair outside 5–40 kWh/100km** so one bad
  reading can't poison the average. The per-session displayed value still renders but is flagged red
  if out of band (see FEAT-5).
- **Recompute** dashboard `total_km` and `avg_efficiency` off the corrected `trip_distance_km` /
  guarded aggregate.
- **qa:** reproduce the 47.55 garbage first; after fix, a decreasing odometer is rejected with the
  Persian message; `total_km` is the monotonic odometer span; `avg_efficiency` lands in ~12–18 on
  good data; a single 480-style outlier no longer moves the average.

### BUG-5 — Unclamp SOH; allow >100% for healthy batteries — MEDIUM

- **Document the rationale** in the SOH explanation text: regenerative braking, downhill driving,

- **Operator flag (no autonomous change):** catalog nominal for the Dongfeng E70 is 47.5 kWh but the

- **security:** none (no auth/secret change). **qa:** with the real data, `/v1/analytics/battery`

### FEAT-1 — Smart SOC prediction (start & end), editable
- **Start SOC default** = previous session's `end_soc` for the same car; if the odometer increased
  since then, subtract estimated consumption:
  `predicted_start = prev_end_soc − (trip_distance_km × estimateConsumption(ctx)/100) / usable_capacity_kwh × 100`.
  Use `estimateConsumption(ctx)` (today = car average) per the Architecture rule.
- **End SOC prediction** = `start_soc + (energy_kwh / usable_capacity_kwh × 100)`, capped at 100.
- Both are **pre-filled, overridable** defaults. If the user's value differs from the prediction by a
  large margin, show a **gentle hint** (not an error; distinct from FEAT-5 warnings).

### FEAT-6 — Duration optional + smart prediction
- Make `duration_minutes` **optional** and move it to the **end** of the form.
- Predict `duration_minutes = energy_kwh / charge_power_kw × 60`; if `charge_power_kw` unknown, use
  the **average charge power from previous sessions at the same location** (FEAT-3 memory).
- Pre-fill the prediction; user can override.

### FEAT-2 — Season-aware expected-consumption band on charts (self-explanatory)
- Add `city` to `user_settings` (dropdown of major Iranian cities) — surfaced in Settings.
- Seasonal expected-consumption multiplier keyed on the **Jalali** month (config object, tunable):
  - Warm (Tir–Shahrivar): **×1.15** (AC)
  - Cold (Dey–Esfand): **×1.20** (heater)
  - Mild (rest): **×1.00**
- On the monthly consumption chart, draw the **expected band/line** from these multipliers (shaded
  "expected range for this season" behind the actual bars) with a legend
  **"محدوده مورد انتظار (فصل گرم/سرد)"**, so a high winter/summer month reads as *normal*, not a
  problem. Multipliers feed `estimateConsumption(ctx)` via `ctx.month`.
- **DESIGN:** the band visualization + warning-row styling (FEAT-5) are design decisions — see Process.

### FEAT-4 — Elevation/regen factor in consumption
- `user_settings.regen_factor` (default 0.10).
- Adjusted consumption = `raw × (1 − regen_factor)`; show **both** raw and adjusted in session detail.
- Elevation heuristic: consecutive sessions at the **same** location → assume no net elevation change;
  **different** locations → soft note "ممکن است اختلاف ارتفاع بر مصرف اثر گذاشته باشد" (do **not**
  attempt correction — no elevation API yet).
- Feed regen into the same `estimateConsumption(ctx)` (`ctx.regenFactor`, `ctx.sameLocationAsPrev`).
