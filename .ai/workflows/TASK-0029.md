## Problem

1. **Phone-only users cannot log in with a password** even if they later set one — the only login path for phone accounts is OTP.
2. **There is no "stay logged in" option** — every session uses a session cookie that clears on browser close, or always uses 30-day persistence (current behaviour). Either extreme; no user choice.
3. **OTP registration requires contact-share first** — user must open the bot and share phone before the web app can send them an OTP. This is the biggest UX friction point for onboarding.
4. **No optional password after OTP registration** — phone-only users are forever passwordless with no opt-in path.
5. **Admins cannot switch OTP delivery strategy** — the contact-share requirement is hardcoded; there is no deep-link alternative.

---

## Goals

- **Part A**: Close the login/registration feature gaps for phone users.
- **Part B**: Add admin-configurable OTP delivery method (Deep Link vs Contact Share).

## Scope In

### Part A — Hybrid Auth Flow

#### A1 — Phone + Password Login

New endpoint **`POST /auth/login/phone`**.

**Request**:
```json
{ "phone": "+989121234567", "password": "s3cr3t", "stay_logged_in": false }
```

**Service**: `AuthService.LoginWithPhone(ctx, phone, password, ip string, stayLoggedIn bool) (access, refresh string, err error)`
- `normalizePhone(phone)` → E.164
- `FindByPhone` → 404 maps to 401 (no enumeration)
- If `PasswordHash == ""` → return `ErrNoPasswordSet` (maps to `400 { code:"NO_PASSWORD_SET", error:"no password set for this account — use OTP to log in" }`)
- `bcrypt.CompareHashAndPassword` → mismatch → 401
- Rate-limit on IP (reuse `loginRateWindow`): same pattern as `Login`
- `issueTokenPair(ctx, userID)` → tokens

**Cookie behaviour** (applies to all login paths — see A2):
- `stay_logged_in=true` → `maxAge = 30 * 24 * 3600`
- `stay_logged_in=false` (default) → `maxAge = 0` (session cookie, browser clears on close)

**Error table**:
| Condition | HTTP | code |
|---|---|---|
| Rate limit | 429 | — |
| Invalid phone format / not found | 401 | `INVALID_CREDENTIALS` |
| No password set | 400 | `NO_PASSWORD_SET` |
| Wrong password | 401 | `INVALID_CREDENTIALS` |

#### A2 — Stay-Logged-In Flag on All Login Paths

Add `stay_logged_in bool` (default `false`) to the request body of:
- `POST /auth/login` (email+password) — already exists, extend the struct
- `POST /auth/login/phone` (new, A1)
- `POST /auth/otp/verify` (phone+OTP login)

Modify `AuthHandler.setRefreshCookie(c *gin.Context, token string, stayLoggedIn bool)`:
```go
maxAge := 0
if stayLoggedIn {
    maxAge = int(refreshTokenTTL / time.Second)
}
c.SetCookie(refreshCookieName, token, maxAge, refreshCookiePath, "", h.isProd, true)
```

The refresh token TTL in Redis is **unchanged** (always `refreshTokenTTL = 30 * 24 * time.Hour`). Only the cookie persistence differs.

Backward-compat: existing `Refresh` and `Logout` handler calls to `setRefreshCookie` pass `false` (Refresh is a transparent rotation, Logout clears with `maxAge=-1`).

#### A3 — Optional Password Setup After OTP Registration

New **authed** endpoint **`POST /v1/account/set-password`**.

**Request**:
```json
{ "password": "newpassword123" }
```

**Service**: `AuthService.SetPassword(ctx context.Context, userID uuid.UUID, password string) error`
- Validate `len(password) >= 8` → `ErrPasswordTooShort`
- `bcrypt.GenerateFromPassword` (cost 12)
- Call new repo method `UserRepository.SetPasswordHash(ctx, id uuid.UUID, hash string) error`
  - `UPDATE users SET password_hash=$2, updated_at=now() WHERE id=$1`
- Returns `nil` on success (idempotent — admin can also re-set a password)

**Response**: `204 No Content` on success. `400` if too short.

#### B4 — Deep Link OTP Flow

When `otp_delivery_method = "deeplink"`, the OTP request flow changes:

**`POST /auth/otp/request`** — new behaviour when method = deeplink AND user has no `chat_id`:

Instead of silently dropping (anti-enum 202) or waiting for a contact share, return a deep link for the user to tap:

```json
HTTP 200
{
  "status": "deep_link",
  "bale_url": "https://ble.ir/BotUsername?start=phone_+989121234567",
  "telegram_url": null    // null if Telegram sender not configured
}
```

If user already has a `chat_id` for the requested platform: send OTP directly (existing path, still returns 202).

If method = `contact_share`: existing anti-enum 202 behaviour unchanged.

**Public endpoint to expose current delivery method** (no auth needed):

**`GET /auth/otp/config`**:
```json
{ "delivery_method": "deeplink" }
```

This lets the frontend know whether to show the deep-link button or the contact-share instruction without requiring the user to be logged in.

#### B5 — Bot Poller: Handle `/start phone_XXXXXX`

The bot poller currently handles:
- `/start <link-token>` → account-link flow
- `/start` (bare) → registration contact-share flow
- contact message → complete link / store registration contact

Add a new case: `/start phone_+989XXXXXXX`

**Bot handler logic**:
```
if strings.HasPrefix(token, "phone_") {
    phone := strings.TrimPrefix(token, "phone_")
    → call cb.HandleDeepLinkOTP(ctx, platform, chatIDStr, phone)
}
```

**`AuthService.HandleDeepLinkOTP(ctx, platform, chatID, rawPhone) error`**:
1. `normalizePhone(rawPhone)` (E.164)
2. Look up user by phone (`FindByPhone`)
   - If found with matching chat_id: this is a login OTP → use `otp:login:platform:phone` key
   - If found but no chat_id or chat_id mismatch: update chat_id, then send login OTP
   - If not found: this is a registration OTP → use `otp:reg:platform:phone` key; also call `StoreRegistrationContact(ctx, platform, chatID, phone)` so `CompleteOTPRegister` can link the account
3. Generate OTP code, store in Redis under the appropriate key (`otpTTL`)
4. Send OTP text to `chatID` via `senderForPlatform(platform)`
5. On success: send confirmation text "✅ کد OTP شما: XXXXXX — ۶۰ ثانیه اعتبار دارد"

Note: `HandleDeepLinkOTP` bypasses the per-IP/per-phone rate limits (it is already rate-limited by the bot's incoming message rate). Add a simple per-phone dedup: if an OTP already exists in Redis for this phone+platform, do not regenerate (return `nil`).

## Security Notes

- `POST /auth/login/phone`: same brute-force mitigation as `POST /auth/login` (IP rate limit, constant-time bcrypt compare, no enumeration of "phone not found" vs "wrong password" — both return 401 `INVALID_CREDENTIALS`). Exception: `NO_PASSWORD_SET` returns 400, not 401, because the caller already proved phone ownership (OTP was issued) and the error is instructional. This is acceptable for this use-case.
- Deep link OTP: the link reveals that the phone is a valid target for OTP delivery. This is an intentional tradeoff — the deep link mode is only meaningful for platforms where the operator controls both the app and the bot. Do not use this mode on public-facing installations where phone number enumeration is a concern.
- `SetPassword`: idempotent and admin-accessible. There is no "current password" check for first-time password set (since `PasswordHash == ""`). For a subsequent change, the user should use a future `change-password` endpoint (out of scope for TASK-0029) that requires the current password. **Scope Out**: password change with old-password verify.
- `stay_logged_in=false` → session cookie: the refresh token still lives 30 days in Redis. If an attacker intercepts the cookie before the browser closes, they can use it. This is no worse than the existing behaviour; it just stops browser-close from being a guarantee of logout.
