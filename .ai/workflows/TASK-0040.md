**Part A — DNS docs.** Added a copy-paste DNS table to `DEPLOY_PRODUCTION.md` (apex/www/mail
A records, MX→mail priority 10, SPF `v=spf1 mx ~all`, DMARC `p=quarantine`, DKIM row deferred

**Part B — Poste.io.** Added the `poste` service to `docker-compose.prod.yml`
(`analogic/poste.io`, hostname `mail.voltanaev.ir`, volume `/var/lib/voltana/mail`).
- Public ports (0.0.0.0): **25, 465, 587, 993** (TLS/MX subset).
- **127.0.0.1-bound**: admin UI **8443** and plaintext **110/143/995** — because Docker
  publishes ports into its own iptables chain *ahead of* UFW (a UFW deny would NOT block a
  published port). Localhost binds are the real guard; documented explicitly.
- `HTTP_PORT=8080`/`HTTPS_PORT=8443` move Poste's web server off 80/443 (nginx owns those).
- `DISABLE_CLAMAV=TRUE` — ClamAV needs ~1 GB; would blow the 4 GB budget alongside
  postgres+redis+api+nginx. Documented as removable on a larger VPS.
- **TLS**: instead of a port-80 fight for Poste's own Let's Encrypt, Poste reuses the certbot
  cert (issued with `mail.voltanaev.ir` as a SAN) mounted read-only; admin points at
  `/etc/letsencrypt/live/voltanaev.ir/{fullchain,privkey}.pem`. Single cert, single renewal.
- `.env.production.example`: SMTP block switched Resend → Poste (`SMTP_HOST=mail.voltanaev.ir`,
  `SMTP_USER/FROM=noreply@voltanaev.ir`, password placeholder).
- `bootstrap-vps-prod.sh`: UFW opens 25/465/587/993, creates `/var/lib/voltana/mail`, certbot
  hint now issues apex+www+mail SANs.

**Part C — prod config.**
- `nginx.prod.conf`: `server_name voltanaev.ir www.voltanaev.ir`; new HTTPS `www → apex` 301
  server (reuses the SAN cert); HTTP server serves ACME for both hosts. envsubst still only
  expands `${DOMAIN}`.
- `.env.production.example`: `DOMAIN`/`APP_URL` → voltanaev.ir; added the **VAPID prod pair**
  block (placeholders + `npx web-push generate-vapid-keys`, "don't reuse dev").
- **No-code findings (documented, not changed):**
  - *CORS / VITE_API_URL*: no CORS middleware exists — the SPA is served by nginx and calls
    the API same-origin; `VITE_API_URL` defaults to empty (same-origin) and must stay unset.
    Documented in the env file + deploy doc. A cross-origin origin would break auth.
  - *PWA manifest*: `public/manifest.json` uses relative paths with `start_url`/`scope` = `/`
    — already domain-agnostic; works under `https://voltanaev.ir` unchanged.
  - *Deep links*: bot deep links are bot-username-based (domain-agnostic); the only
    domain-driven URL is the email verify link, built from `APP_URL` — set correctly in prod.
