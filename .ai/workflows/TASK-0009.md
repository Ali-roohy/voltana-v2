## Goal

Close the email-verification gap left open by TASK-0002: actually send verification
emails on registration and refuse login for accounts that have not verified their
email. This was deferred from TASK-0002 (note **N2**) — the `email_verification_tokens`
table exists (migration `000002`) but is unused, and `is_email_verified` is never
enforced at login.

---

## Scope In

- **Login gate** — `POST /auth/login` returns **403** with a clear, machine-readable
  body (e.g. `{"error":"email not verified","code":"EMAIL_NOT_VERIFIED"}`) when the
  user exists, the password is correct, but `is_email_verified = false`. Wrong
  credentials must still return 401 (do not leak verification state for bad passwords).
- **SMTP dispatch** — implement an email sender using the existing
  `SMTP_HOST / SMTP_PORT / SMTP_USER / SMTP_PASSWORD / SMTP_FROM` env vars
  (already placeholdered in `.env.example`).
- **On register** — generate a verification token, store its **SHA-256 hash** in
  `email_verification_tokens` (raw token never persisted) with an expiry, and email the
  user a verification link containing the raw token.
- **`POST /auth/verify-email`** — accept `{ "token": "..." }`, hash it, look up an
  unexpired row, set `users.is_email_verified = true`, and delete/invalidate the token.
- **Resend** — `POST /auth/resend-verification` (rate-limited) to re-issue a token for
  an unverified, authenticated-by-email request.
- Architecture: email sending behind an interface (`service.Mailer`) injected into the
  auth service, so it is mockable in unit tests and SMTP is not called in tests.

## Scope Out

- Verification UI / email templates styling (frontend — handled with **TASK-0006**).
- Password-reset emails (separate later task).
- Social login (Phase 4).

## API Contract (architect — 2026-05-30)

Both endpoints are **public** (no `Authorization` header). Error envelope follows the
existing auth handlers, extended with a machine-readable `code`:
`{ "error": "<human message>", "code": "<MACHINE_CODE>" }`.

**Verification token** (shared by both endpoints): opaque, ≥256-bit CSPRNG, base64url-
encoded, single-use, TTL **24h**. Only its **SHA-256 hash** is stored in
`email_verification_tokens.token_hash`; the raw token travels only in the emailed link
(`<APP_URL>/verify-email?token=<raw>`) and the verify request body. Never logged.

### POST /auth/verify-email

Consume a verification token and mark the account verified.

Request body:
```json
{ "token": "<raw verification token>" }
```

| Code | When | Body |
|------|------|------|
| `200 OK` | token valid & unexpired → `is_email_verified=true`, token deleted (single-use) | `{ "message": "email verified" }` |
| `200 OK` | account already verified (idempotent replay of a now-consumed flow) | `{ "message": "email already verified" }` |
| `400 Bad Request` | missing/malformed body (fails `binding:"required"`) | `{ "error": "invalid request body", "code": "INVALID_REQUEST" }` |
| `400 Bad Request` | token unknown / expired / already used (do **not** distinguish — anti-enumeration) | `{ "error": "invalid or expired token", "code": "INVALID_VERIFICATION_TOKEN" }` |
| `429 Too Many Requests` | per-IP limit exceeded (see below) | `{ "error": "too many attempts, try again later", "code": "RATE_LIMITED" }` |

- Token compared by hashing the input and looking up `token_hash`; treat the lookup as
  constant-time-ish (single indexed equality, no early "user exists" branch).
- On success the row is **deleted** in the same transaction that flips `is_email_verified`.

**Rate limit:** per IP — **20 / 15 min**, key `ratelimit:verify:<ip>` (reuse the existing
fixed-window Lua limiter). Tokens are high-entropy so brute force is infeasible; this cap
is abuse insurance only.

### POST /auth/resend-verification

Re-issue a verification email for an unverified account.

Request body:
```json
{ "email": "<email>" }
```

| Code | When | Body |
|------|------|------|
| `202 Accepted` | **always** for a well-formed request — existing+unverified (new token issued, prior tokens for that user invalidated, email sent), existing+already-verified (no-op), or unknown email (no-op) | `{ "message": "if the account exists and is unverified, a verification email has been sent" }` |
| `400 Bad Request` | missing/malformed email (fails `binding:"required,email"`) | `{ "error": "invalid request body", "code": "INVALID_REQUEST" }` |
| `429 Too Many Requests` | per-IP **or** per-email limit / cooldown exceeded | `{ "error": "too many requests, try again later", "code": "RATE_LIMITED" }` |

- **Anti-enumeration:** the `202` body and timing must be identical whether or not the
  email exists or is already verified. Do the (no-op) work paths so latency is uniform.
- Issuing a new token invalidates the account's previous unexpired verification tokens
  (delete-then-insert) so only the latest link works.

**Rate limit (two dimensions + cooldown):**
- per IP — **5 / hour**, key `ratelimit:resend:ip:<ip>`.
- per account — **3 / hour**, key `ratelimit:resend:email:<sha256(lowercased email)>`
  (hash the email so raw addresses are never used as Redis keys / logged).
- **Cooldown** — minimum **60 s** between successful sends to the same email; a request
  inside the cooldown returns `202` (no new email) to preserve anti-enumeration.
- All counters use the existing fixed-window Lua limiter; the per-email key protects a
  victim from email-bombing, the per-IP key protects the SMTP relay.

> verification link, alongside the already-present `SMTP_*` vars.

## Acceptance Criteria

- [ ] Unverified user with correct password → `POST /auth/login` returns **403** with
      `code: EMAIL_NOT_VERIFIED`; verified user logs in normally (200).
- [ ] Wrong password still returns **401** regardless of verification state.
- [ ] Registration stores only the **SHA-256 hash** of the verification token (raw token
      never in DB or logs) and sends an email via SMTP.
- [ ] Verification token has a finite TTL (recommend 24h); expired/used tokens rejected.
- [ ] `POST /auth/verify-email` flips `is_email_verified` and consumes the token.
- [ ] `POST /auth/resend-verification` is rate-limited (reuse the existing limiter).
- [ ] SMTP failures are logged without leaking credentials; registration still succeeds
      (email send is best-effort / retryable) — decide and document the chosen behavior.
- [ ] `Mailer` is an injected interface; `go test ./internal/service/...` passes with a
      mock mailer (no real SMTP in tests).

---

## Security Notes (for security review)

- Verification tokens: store SHA-256 hash only; compare in constant time on verify.
- Never log raw tokens, SMTP password, or full recipient address (mask emails as in
  `auth_handler.go:maskEmail`).
- The 403 gate must not reveal verification state on **wrong** passwords (avoid
  account-enumeration) — only gate after a successful credential check.
- Rate-limit `resend-verification` to prevent email-bombing a victim address.

## Developer Implementation (2026-06-02) — DONE, → REVIEW

Implemented the full email-verification gate (backend) + bug #7 UI (frontend). Behind a `service.Mailer`
interface so SMTP is never reached in unit tests. **No new migration** — `email_verification_tokens`
(000002) already fits (`token_hash VARCHAR(64)` = SHA-256 hex).

**Backend — created:**
- `internal/repository/verification_repo.go` — `VerificationTokenRepository`:
  `ReplaceVerificationToken` (delete-then-insert in a txn → one outstanding token) and
  `ConsumeVerificationToken` (single txn: `SELECT … FOR UPDATE` by hash + unexpired → delete user's
  tokens + flip `users.is_email_verified`; returns `alreadyVerified`; `ErrNotFound` → 400).
- `internal/mailer/mailer.go` — `SMTPMailer` (net/smtp, `SMTP_*`) + `LogMailer` (dev; never logs the
  token/URL or full address). Satisfies `service.Mailer` structurally (no import of service).

**Backend — changed:**
- `internal/service/auth_service.go` — `Mailer` interface; new errors `ErrEmailNotVerified` /
  `ErrInvalidVerificationToken`; constructor now takes `(users, verifications, tokens, mailer, appURL,
  jwtSecret)`. **Register** issues a token (256-bit CSPRNG base64url; only SHA-256 hex stored) + emails the
  link — **best-effort** (failures logged by user ID, registration still succeeds). **Login** returns
  `ErrEmailNotVerified` **only after** a successful password check (wrong password still
  `ErrInvalidCredentials` — no enumeration). New `VerifyEmail` (per-IP 20/15m) and `ResendVerification`
  (per-IP 5/h + per-email 3/h on `sha256(lowercased email)` keys + 60s cooldown via max-1/60s; always

  `APP_URL`; registered the two routes. `.env.example` — added `APP_URL`.

**Backend — tests** (`auth_service_test.go`): added mock verification repo (mirrors the atomic flip) +
