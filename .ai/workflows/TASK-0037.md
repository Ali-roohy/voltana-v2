## FEAT-2 — Full PWA install support

  - valid `manifest.json` / `manifest.webmanifest`: `name`, `short_name`, icons **192 + 512**
    (+ maskable), `theme_color`, `background_color`, `display: standalone`, `start_url`, `dir: rtl`, `lang: fa`;

    already installed / iOS shows manual hint).

- **AC**: Lighthouse PWA installability criteria pass (or, no browser on host: all criteria
  verified individually — manifest fetch 200 + valid JSON, icons reachable, SW registered in
  build output, served over the existing origin). Install button appears and triggers the prompt.

## FEAT-4 — Per-user data backup & restore (Settings)

- «پشتیبان‌گیری و بازگردانی» section at the end of Settings.
- **Export**: `GET /v1/account/export` → JSON download of the user's own data: cars, charging
  sessions, settings, battery snapshots (+ export schema version + exported_at).
- **Import**: upload that JSON → server-side validation → restore into the user's account.
  Conflict strategy: **replace own data after explicit confirmation dialog** (type-to-confirm not
  required; a clear destructive-action dialog is).
- **Security (review stage required)**: strictly scoped to the authenticated user; **all IDs
  re-mapped on import** (never trust embedded user_id/car_id/session ids); size limit on upload;
  reject unknown schema versions cleanly.
- **AC**: export → wipe-and-import round-trip restores cars/sessions/settings/snapshots with
  identical counts + costs; importing a file with foreign IDs cannot touch another user's rows
  (cross-user attempt proves isolation); invalid JSON → 400 with a clear message.

## FEAT-5 — Self-delete account (Settings)

- «حذف حساب کاربری» at the end of Settings for every user EXCEPT the permanent first admin.
- Flow: type-to-confirm dialog → `DELETE /v1/account` → cascade all user data (cars, sessions,
  snapshots, settings, refresh tokens) → logout + redirect.
- **Backend guard**: refuse deletion if user is the first/last admin — reuse `ErrLastAdmin`
  guards from TASK-0028. UI hides the section for that user; backend still enforces.
- **AC**: non-admin self-delete removes all rows (DB-verified) + revokes refresh token + logs out;
  last-admin delete → 4xx with clear error AND section hidden in UI; security review passes.

## FEAT-6 — Per-user electricity rates with historical rate snapshot (CRITICAL — money)

- **Admin defaults**: Admin Settings gains "default rates" (peak/mid/offpeak) stored in
  `system_settings` (TASK-0029 migration 000013 infra). Remove hardcoded rate defaults from the
  user_settings creation path/migration defaults.
- **Registration copy**: new user registration copies the CURRENT admin defaults into the new
  user's `user_settings`.
- Users keep editing their own rates in Settings (already exists).
- **CRITICAL — rate snapshot**: new migration adds `rate_peak_at_time, rate_mid_at_time,
  rate_offpeak_at_time` to `charging_sessions`. Every session **snapshots the owner's rates at
  creation time**. Cost calculation and ALL aggregate analytics (dashboard totals, monthly trend,
  TOU breakdown, avg cost/session) use the **snapshotted** rates. Changing rates later must NOT
  retroactively change old sessions' costs.
- **Migration backfill**: existing sessions' snapshot columns backfilled from the owner's current
  rates. Down migration reverses clean.
- **Frontend note**: `lib/cost.ts` currently computes costs client-side from current settings —
  it must consume per-session snapshot rates from the API instead (audit every call site:
  TOUBreakdown, dashboard cards, monthly trend, session cards).
- **qa scenario (operator-specified, mandatory)**: create session 1 → change rates → create
  session 2 → each session keeps its own cost; dashboard totals = sum of snapshotted costs.
- **AC**: the qa scenario passes live; admin default change affects only NEW registrations'
  starting rates; backfill correct for existing rows; `go test` covers snapshot-vs-current.
