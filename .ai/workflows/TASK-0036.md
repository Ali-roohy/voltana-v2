## BUG-1 — Full name not saved on email registration

- **Report**: user entered full name during email registration; Admin → User Management shows empty name column.
- **PM pre-triage**: `auth_handler.go` `registerRequest` has only `email`+`password`; handler calls
  `auth.Register(ctx, req.Email, req.Password)` — `full_name` is dropped server-side even if sent.
  Check the frontend register form too (does it send `full_name` at all?).
- **qa repro**: register via UI/curl with full name → `GET /v1/admin/users` → name empty.
- **Fix**: persist end-to-end: frontend form → `POST /auth/register {full_name}` (optional field)
  → `users.full_name` → admin list + `/v1/me`.
- **AC**: new email registration with a name shows that name in admin user list and `/v1/me`.
  Registration without a name still works (field optional).

## BUG-2 — Phone not saved on email registration

- **Report**: phone entered during email registration; admin phone column empty.
- Same end-to-end treatment as BUG-1 for `phone` (normalize via `normalizePhone`; respect
  `uq_users_phone` — duplicate phone on email-register must produce a clean 409, not a 500).
- **AC**: email registration with phone persists normalized E.164 phone; visible in admin list;
  duplicate phone → 409.

## BUG-3 — Admin flag lost after second registration (CRITICAL)

- **Report**: first user was admin; after a second user registered, NO user was admin. Even
  `UPDATE users SET is_admin = true` didn't restore admin behavior in the UI.
- **PM pre-triage**: the INSERT (`user_repo.go:55` and `:86`) grants admin via
  `NOT EXISTS (SELECT 1 FROM users)` — it cannot demote existing rows. Suspects, in order:
  1. **Stale frontend cache**: TanStack `useMe` cache not invalidated/reset on login/logout —
     admin panel gating reads a stale `/v1/me`. (Would also explain "SQL update + re-login didn't help".)
  2. Registration flow side effects (e.g. `AdminUpdate` COALESCE path, bot-link `UpdateBotLink`)
     touching the wrong row.
  3. Two distinct users tables/DBs in play (operator pointed psql at the wrong DB).
- **qa repro (mandatory, exactly as operator specified)**: clean state → register A (verify
  `is_admin=true` in DB AND admin UI visible) → register B → verify A still admin (DB + UI),
  B not admin. Then SQL-update test: set is_admin manually → re-login → admin panel appears.
- **Expected behavior (locked)**: admin is granted ONLY to the very first registered user and
  never changes automatically; registration must NEVER touch existing users' `is_admin`.
- **AC**: qa's exact scenario passes end-to-end; root cause documented in this file with evidence.

## BUG-5 — OTP "awaiting code" status design is wrong (DESIGN + FIX)

- **Report**: immediately after phone submit, UI shows «در انتظار دریافت کد از بله/تلگرام» with a

- **Correct flow**:
  (a) after phone submit show the «باز کردن بله برای دریافت کد» button with neutral instructions,
  **no countdown, generous/no timeout**;
  (b) only after bot interaction is detected (poll sees contact/deep-link consumed) switch to
  "awaiting code" and start the OTP timer.
- **DESIGN PHASE (Fable)**: redesign this step's states / copy / timeouts before implementation.
  Design output goes in this file under `## BUG-5 Design (Fable)`.
- **Scope includes** the operator's uncommitted Android `intent://` deep-link WIP in `BotOTPTab`
  (adopted as-is unless the design changes the button).
- **AC (qa)**: status text and timer change ONLY after the bot detects the user; no premature
  countdown; deep-link button works (Android intent path code-reviewed).

## BUG-8 — OTP Test Panel: add «تست ارتباط» (connectivity test) per platform

- New backend endpoint (**AdminOnly**): `POST /v1/admin/test-bot-connection {platform}` →
  server-side `getMe` call with the **real bot token from env** (NEVER hardcoded, never sent to
  the client):
  - Bale: `https://tapi.bale.ai/bot<BALE_BOT_TOKEN>/getMe`
  - Telegram: `https://api.telegram.org/bot<TELEGRAM_BOT_TOKEN>/getMe`
- Response: `{ok, bot_username, latency_ms}` or upstream error (sanitized — no token echo).
- UI: button next to each «تست ارسال»; green check + bot username on success, red badge + error on failure.
- **AC**: admin-only (403 non-admin); works per platform; missing env token → clean error, not a panic;
  token never appears in response, logs, or frontend; timeout-bounded HTTP call.

## BUG-5 Design (Fable) — Deep-link OTP step state machine

**Root cause recap**: deep-link sessions write no Redis key at request time, so the status poll
can only answer "otp_sent" or "expired" — there is no representable "user hasn't opened the bot
yet" state. The frontend additionally renders an "awaiting code" spinner the moment the deep link
appears.

### States (deep-link mode)

| State | Server signal | UI | Timer |
|---|---|---|---|
| **S1 ready** (default after phone submit) | `awaiting_bot` (new) | Neutral copy: «روی دکمه زیر بزنید؛ ربات بله باز می‌شود و کد برای شما ارسال می‌شود.» + button «📲 باز کردن بله برای دریافت کد» + muted hint «پس از باز کردن ربات، کد به‌صورت خودکار ارسال می‌شود» — **no spinner, no countdown** | none |
| **S2 code sent** | `otp_sent` (existing) | «✅ کد ارسال شد — کد ۶ رقمی را وارد کنید» + OTP input + 60 s countdown + resend (after cooldown) | starts now |
| **S3 expired** | `expired` (now means *real* abandonment only) | toast «مهلت دریافت کد تمام شد — دوباره امتحان کنید» → back to phone step | — |

### Server changes

1. `requestOTPDeeplink` (deep-link branch, no chat_id): set marker
   `otp:dlpending:<platform>:<E164>` with **`deeplinkPendingTTL = 15 min`** (the "generous
   timeout" — long enough to install the app, log into Bale, etc.).
2. `CheckContactShareStatus` order: sentKey → active `otp:reg:/otp:login:` → **dlpending marker →
   `"awaiting_bot"`** → contact_share pendingKey logic (unchanged) → `"expired"`.
3. `HandleDeepLinkOTP` deletes the marker when it dispatches the code (state cleanliness; the
   active-OTP check already supersedes it).

### Frontend changes (`BotOTPTab`)

- Poll handler: `awaiting_bot` → keep waiting (no-op). `otp_sent`/`expired` unchanged — but
  `expired` is now only emitted at true abandonment, fixing the 3-second kick-out.
- Deep-link branch render: S1 shows the neutral hint **without** the spinner/«در انتظار ارسال
  کد...» line; after `timerStarted` (S2) show «✅ کد ارسال شد» + countdown (existing timer UI).
- Keep the operator's Android `intent://` handler (adopted WIP).
- contact_share mode untouched (its pendingKey/instruction flow was already correct).
