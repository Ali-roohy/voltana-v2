## 1. Versioning Strategy

- **Scheme:** [Semantic Versioning](https://semver.org) `vMAJOR.MINOR.PATCH` (git tags prefixed `v`).
- **Current version:** **`v0.3.0`** (Phase 1 + Phase 2 complete; Phase 3 partially done — infra hardening
  TASK-0014 + map/stations TASK-0013).
- **Single source of truth:** a root **`VERSION`** file containing the bare number (e.g. `0.3.0`), matched by an
  annotated git tag `v0.3.0`. Keep `voltana-web/package.json` `version` in sync on each bump (optional, nice for
  the build banner). The Go API may surface it later via `/health` using a build-time ldflag — out of scope here.
- **When to bump (pre-1.0 policy):**

  | Part | Bump when | Examples |
  |---|---|---|
  | **MAJOR** (`x`) | Reserved for the **v1.0.0 commercial release**. While `0.y`, breaking changes ride a MINOR bump (per SemVer §4) but must be called out in `CHANGELOG.md`. | First stable public release |
  | **MINOR** (`y`) | A **feature task closes** (new endpoints, new backward-compatible migration, a phase milestone). | TASK-0013 stations → `0.3.0`; OBD feature → `0.4.0` |
  | **PATCH** (`z`) | Bug fix, docs, infra/CI tweak, dependency bump — **no new feature**. | A station validation fix → `0.3.1` |

- **Release procedure (document in `CONTRIBUTING.md` or the PR body):** update `VERSION` + `CHANGELOG.md`
  (move `Unreleased` → the new version + date), merge to `main`, then
  `git tag -a v0.3.0 -m "v0.3.0" && git push origin v0.3.0`. Tag **after** CI is green on `main`.

> **Decision flag for operator:** the repo already has a lowercase **`changelog.md`** (task-oriented). GitHub
> conventionally surfaces **`CHANGELOG.md`** (version-oriented, "Keep a Changelog" format). Recommendation:
> **promote `changelog.md` → `CHANGELOG.md`** restructured by version, and keep the per-task detail as the body
> of each version section. Do **not** keep two divergent changelogs. (See §2.5.)

## 3. CI Pipeline — `.github/workflows/ci.yml`

Triggers on push **and** PR to `main`. Two independent jobs; **no deployment**.

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  api:
    name: Go API — build · vet · test
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: voltana-api
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version: '1.22'
          cache-dependency-path: voltana-api/go.sum
      - run: go build ./...
      - run: go vet ./...
      - run: go test ./...

  web:
    name: Frontend — typecheck · build
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: voltana-web
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: npm
          cache-dependency-path: voltana-web/package-lock.json
      - run: npm ci
      - run: npx tsc --noEmit
      - run: npm run build
```

> **Prereq status (verified 2026-06-03):** `voltana-web/package-lock.json` **is present and tracked**, and
> `voltana-api/go.sum` is committed — so `npm ci` and the Go module cache both work out of the box. No action
> needed unless the lockfile drifts.

## 4. Branch Protection Rules (document + apply on `main`)

Apply via **Settings → Branches → Add rule** (or `gh api`, below). Document these in `CONTRIBUTING.md`:

- **Protect `main`** — no direct pushes; all changes via PR.
- **Require a pull request before merging** (≥ **1** approval; for a solo maintainer, CODEOWNERS review still
  enforces self-review discipline — relax to 0 approvals only if working solo and note it).
- **Require status checks to pass before merging**, and **require branches to be up to date**:
  required checks = **`Go API — build · vet · test`** and **`Frontend — typecheck · build`** (the two CI jobs).
- **Require conversation resolution before merging.**
- **Do not allow force pushes**; **do not allow deletions** of `main`.
- (Optional) **Require linear history** (squash or rebase merges only).

Apply with the GitHub CLI:
```bash
gh api -X PUT repos/Ali-roohy/voltana-v2/branches/main/protection \
  -H "Accept: application/vnd.github+json" \
  -f 'required_status_checks[strict]=true' \
  -f 'required_status_checks[checks][][context]=Go API — build · vet · test' \
  -f 'required_status_checks[checks][][context]=Frontend — typecheck · build' \
  -F 'enforce_admins=true' \
  -F 'required_pull_request_reviews[required_approving_review_count]=1' \
  -F 'restrictions=' \
  -F 'allow_force_pushes=false' \
  -F 'allow_deletions=false'
```
> The required-check **contexts must exactly match the CI job `name:` values** in §3. If you rename the jobs,
> update the protection rule too. Status checks can only be marked "required" after they've run at least once,
> so open the first PR, let CI run, then enable the rule.
