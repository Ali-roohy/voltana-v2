## Cost-model decision (resolved 2026-05-31)
The existing table had a single `kwh_charged`; the architecture doc's cost model needs a per-period
energy split. **Chosen (operator): doc-faithful per-period columns.** Migration `000004` adds
`energy_peak_kwh`/`energy_mid_kwh`/`energy_offpeak_kwh`; `kwh_charged` is retained as the grand total.
Server-side cost = `peak_kwh·peak_rate + mid_kwh·mid_rate + offpeak_kwh·offpeak_rate`, using the three
`user_settings` rates. If `cost` is supplied it is kept as-is; if no period energy is supplied, cost
stays NULL (nothing to compute).

### Migration schema (battery_health_snapshots — HISTORY table, one row per recompute)
```sql
CREATE TABLE battery_health_snapshots (
    id                     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    car_id                 UUID NOT NULL REFERENCES cars(id)  ON DELETE CASCADE,
    user_id                UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    soh_pct                DECIMAL(5,2) NOT NULL,   -- clamped (0,100]
    estimated_capacity_kwh DECIMAL(6,2) NOT NULL,
    nominal_capacity_kwh   DECIMAL(6,2) NOT NULL,   -- ev_models.battery_capacity_kwh at compute time
    sample_session_count   INT          NOT NULL,
    confidence             VARCHAR(10)  NOT NULL,   -- 'low' | 'medium' | 'high'
    method                 VARCHAR(20)  NOT NULL DEFAULT 'delta_soc',
    computed_at            TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE INDEX bhs_car_computed_idx ON battery_health_snapshots(car_id, computed_at DESC);
```

### Recompute trigger — **NO asynq** (architect ruling)
`asynq` is not a dependency and there is no worker process/container; adding one to the
2 vCPU / 4 GB host is unjustified for a single cheap aggregate query. **Recompute synchronously
in `analytics_service` on charging-session write**, via a fire-and-forget goroutine with
**per-car dedup/singleflight** so rapid writes coalesce. Trigger on session **create, update,
AND delete** (not just POST — edits change history). Revisit a real async broker only if a later
task proves it necessary.

### Algorithm (delta-SOC)
- **Qualifying session** (all required): `kwh_charged > 0`, both `start_soc`/`end_soc` present,
  `end_soc > start_soc`, and **Δsoc ≥ 25** (recommend 40). `start_soc`/`end_soc` are INT (whole %),
  so small swings are too noisy to use.
- **Per-session capacity** (account for charging losses — wall energy ≠ pack energy, else SOH > 100%):
  `cap_i = (kwh_charged_i × η) / (Δsoc_i / 100)`, with `η = 0.88` as a named constant `chargingEfficiency`.
- **SOH** = `100 × weightedAvg(cap_i) / nominal_capacity_kwh`, **weighted by Δsoc_i** (larger swing →
  lower relative error), then **clamped to (0,100]**.
- **Minimum 5 qualifying sessions** to emit a `confidence:'high'`-eligible snapshot; map sample
  count + Δsoc spread → `confidence`. Fewer than 5 qualifying, OR `ev_model_id IS NULL`
  (nullable FK → no nominal capacity) → **do not fabricate**: return insufficient-data (see API).

### 1. `GET /v1/analytics/dashboard` response (lifetime, all cars)
```json
{

  "total_cost": 678900,
  "total_km": 15000,
  "avg_kwh_per_100km": 16.20,
  "session_count": 42
}
```

  cost is **already persisted server-side** at write (client value or TOU-computed from rates), so no

  meaningless number before any odometer is set).
- **Approximation, documented:** odometer is a lifetime current value (defaults 0 on car create) and charging
  kWh includes charging losses, so efficiency is a rough lifetime estimate, not a trip metric. A
  date-windowed efficiency is **out of scope** — it needs a per-session odometer column (future schema work).

```json
] }
```
