## Security Review (2026-05-30) — **FAIL** (superseded by fix pass above)

Reviewed: auth_service.go, auth_handler.go, middleware/auth.go, redis_token_store.go,
user_repo.go, migrations/000002. Two findings confirmed live against the running stack.

### S1 — [HIGH] Refresh-token revocation is not atomic (concurrent replay forks the session)
- Where: `service/auth_service.go` `Refresh()` lines ~109–124 (Load → Delete → reissue) +
  `repository/redis_token_store.go` `DeleteRefreshToken` (uses `Del(...).Err()`, never checks
  the delete count).
- Why it fails: `LoadRefreshToken` then `DeleteRefreshToken` is a TOCTOU sequence. Two
  concurrent requests with the same token both pass the Load check before either Delete runs,
  so both reach `issueTokenPair`. Directly violates the task's security note
  *"invalidate old token BEFORE issuing new one (prevent replay)."* Sequential replay is
  blocked, concurrent replay is not, and there is no reuse-detection to revoke the family.
- Evidence: 20 concurrent `POST /auth/refresh` with one token → **2× HTTP 200** (token forked),
  18× 401.
- Required fix: make consume-old atomic. Replace the Load+Delete pair in `Refresh` with a
  single atomic op — go-redis v9 `rdb.GetDel(ctx, "refresh:"+jti)` — and treat a missing key
  as `ErrTokenRevoked`. (Alternatively a Lua script / `WATCH`+`MULTI`.) Verify `GetDel`
  returns the expected userID before issuing the new pair.

### S2 — [HIGH] Login rate limit is bypassable — keyed on spoofable X-Forwarded-For
- Where: `service/auth_service.go` `Login()` uses `ip` from `c.ClientIP()`
  (`handler/auth_handler.go` line ~85); `cmd/server/main.go` `gin.New()` sets **no trusted
  proxies**, so Gin returns the leftmost (client-controlled) X-Forwarded-For value.
- Why it fails: only nginx is exposed, but nginx appends to whatever XFF the client sends, and
  Gin (no trusted proxies configured) trusts the leftmost entry. An attacker rotates the header
  to get a fresh counter every request. Defeats AC *"Rate limit: 10 login attempts / 15 min
  per IP."*
- Evidence: fixed spoofed `X-Forwarded-For: 203.0.113.7` → `429` at attempt 11 (limit works
  per-key); rotating `X-Forwarded-For: 198.51.100.$i` per request → **never 429** (12×401).
  Redis showed one `ratelimit:login:<spoofed>` key per value.
- Required fix: configure trusted proxies so `ClientIP()` resolves to nginx's appended real IP
  — e.g. `r.SetTrustedProxies([]string{"172.16.0.0/12"})` (the Docker bridge subnet) in
  `main.go`, or key the limiter on the `X-Real-IP` nginx already sets from `$remote_addr`.
  (Note: `main.go` was not in the listed review files but is the locus of this fix.)

### S3 — [LOW] CheckRateLimit INCR+EXPIRE not atomic
- `repository/redis_token_store.go` lines ~41–50: if the process dies between `Incr` and
  `Expire` (or Expire errors after a successful Incr), the key persists with no TTL → that IP
  is locked out indefinitely. Recommend a small Lua script (INCR + conditional EXPIRE) or
  `SET key 1 EX <window> NX` + `INCR` pattern. Not blocking on its own.

### S4 — [LOW/INFO] HS256 not strictly enforced
- `parseToken` (auth_service.go ~191) accepts any `*jwt.SigningMethodHMAC` (HS256/384/512), not
  HS256 only. No exploit — all HMAC algs use the same secret and `alg=none`/RSA confusion is
  already blocked — but add `jwt.WithValidMethods([]string{"HS256"})` to match the spec exactly.

### Passing checks (no action)

- JWT secret from env only, no fallback. ✅
- Refresh tokens stored in Redis with TTL (~30d verified). ✅
- Cookie: `HttpOnly` always, `Secure` gated on `isProd`, `SameSite=Strict`, `Path=/auth`. ✅
- Access/refresh `type` claim separation (refresh rejected as access). ✅
- Log hygiene: no password / token / Authorization values logged; email masked. ✅
