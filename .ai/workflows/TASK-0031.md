## Problem

TASK-0025 shipped the initial VPS deployment layer (bootstrap, deploy script, nginx, systemd, DEPLOY.md).
What is still missing for a production-hardened deployment:

1. **Database backups** — no automated pg_dump; a disk failure or mis-migration loses all data.
2. **Bind-mount postgres volume** — docker named volume hides the data directory; a bind mount at `/var/lib/voltana/postgres` makes backups trivial and location explicit.
3. **Ubuntu 24.04 bootstrap** — TASK-0025's `bootstrap-vps.sh` targets 22.04; new VPS provisioning should target 24.04 LTS.
4. **Production `.env` template** — `.env.example` blends dev and prod; operators need a clean prod-only template.
5. **Health check script** — no automated `/health` monitoring or alerting.
6. **Comprehensive `DEPLOY_PRODUCTION.md`** — `docs/DEPLOY.md` covers basic steps; a production guide needs backups, monitoring, restore procedures, scaling path.

## Scope In

### 1. `scripts/bootstrap-vps-prod.sh`

Ubuntu 24.04 LTS bootstrap (idempotent, safe to re-run). Extends TASK-0025's script with:
- Version check: warn if not Ubuntu 24.04
- Create `/var/lib/voltana/postgres` and `/var/lib/voltana/backups` with `voltana:voltana` ownership
- Install `awscli` (optional, for S3 backups)
- Install the systemd backup timer units
- Print next-steps referencing `docs/DEPLOY_PRODUCTION.md`

### 2. `docker-compose.prod.yml` update

Override postgres to use a bind mount instead of a named volume:
```yaml
services:
  postgres:
    volumes:
      - /var/lib/voltana/postgres:/var/lib/postgresql/data
```
Explicit host path makes `pg_dump` location transparent and simplifies backup scripts.

### 3. `scripts/backup-db.sh`

Daily pg_dump automation:
- `pg_dump` via `docker exec voltana-postgres` → gzip → `/var/lib/voltana/backups/voltana_YYYYMMDD_HHMMSS.sql.gz`
- 7-day local retention (`find -mtime +7 -delete`)
- Optional S3 upload when `AWS_S3_BACKUP_BUCKET` is set (uses `aws s3 cp`)
- Logs to `/var/lib/voltana/backups/backup.log`
- Exit non-zero on failure (so systemd marks the unit failed)

### 4. `infra/systemd/voltana-backup.service` + `voltana-backup.timer`

Systemd timer triggers `backup-db.sh` daily at 03:00 UTC.
- `voltana-backup.service` — `Type=oneshot`, `User=voltana`, `ExecStart=bash /opt/voltana/scripts/backup-db.sh`
- `voltana-backup.timer` — `OnCalendar=*-*-* 03:00:00 UTC`, `Persistent=true` (fires once after boot if it missed the window)

### 5. `.env.production.example`

Production-only template (no dev fields, no MailHog):
```

APP_ENV=production
APP_URL=https://voltana.example.com
PORT=9090
POSTGRES_DB=voltana
POSTGRES_USER=voltana_user
POSTGRES_PASSWORD=<generate: openssl rand -hex 32>
JWT_SECRET=<generate: openssl rand -hex 32>
JWT_ACCESS_TTL=15m
JWT_REFRESH_TTL=720h
SMTP_HOST=smtp.resend.com
SMTP_PORT=587
SMTP_USER=resend
SMTP_PASSWORD=<resend api key>
SMTP_FROM=noreply@yourdomain.com
BALE_BOT_TOKEN=<bale botfather token>
BALE_BOT_USERNAME=<bale_bot_username>
TELEGRAM_BOT_TOKEN=<telegram botfather token>
TELEGRAM_BOT_USERNAME=<telegram_bot_username>
# Optional S3 backups
# AWS_S3_BACKUP_BUCKET=your-bucket-name
# AWS_ACCESS_KEY_ID=...
# AWS_SECRET_ACCESS_KEY=...
# AWS_DEFAULT_REGION=us-east-1
```

### 6. `scripts/healthcheck.sh`

Curl `/health`; on failure:
- Log to `/var/log/voltana-health.log` with timestamp
- Optionally send a Bale/Telegram alert if `ALERT_BOT_TOKEN` + `ALERT_CHAT_ID` are set
- Exit non-zero (cron or systemd sees the failure)

### 7. `docs/DEPLOY_PRODUCTION.md`

Comprehensive production operations guide:
- Full deployment checklist (from VPS provisioning to first live request)
- Environment variables reference
- TLS certificate acquisition + auto-renewal
- Backup setup + verification + restore procedure
- Health check + monitoring setup
- Log aggregation (`docker compose logs --follow`)
- Troubleshooting table (extended from DEPLOY.md)
- Update / rollback procedure
- Scaling path notes (vertical scale, postgres replication)
