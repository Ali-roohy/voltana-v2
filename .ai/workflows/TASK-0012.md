## Goal

Make the charging history actually browsable: add a **date-range filter** to the sessions
list (the API already supports `?from` / `?to`) and a **tap-to-expand detail view** for each
session card so users can drill into the full record without leaving the list. Medium impact,
low effort ‚Äî **API already supports the filter**; the work is frontend wiring + a detail UI.

Source of the proposal: researcher report, top-3 (Med impact / Low effort).

---

## Background / Grounding (already in the codebase)

- **API supports it.** `GET /v1/charging-sessions` (handler `charging_handler.go`,
  `parseChargingFilter`) already reads `?car_id`, `?from`, `?to` (**RFC3339**, malformed ‚Üí
  `400 INVALID_REQUEST`) plus pagination, returning `{items, limit, offset, total}` (limit ‚â§ 100).
- **Frontend does NOT pass them yet.** `features/charging/api.ts ‚Üí listChargingSessions()`
  hardcodes `"/v1/charging-sessions?limit=100"` with **no** filter params, and
  `hooks.ts ‚Üí useChargingSessions()` uses a static query key `["charging-sessions"]`. This task
  must extend both to accept an optional filter and **include the filter in the query key** so
  TanStack Query caches per-range correctly (no `useEffect` fetching ‚Äî ADR-002).
- The page (`pages/Charging.tsx`) already has a **car** filter (`selectedCarFilter`) and a
  `JalaliDatePicker` component is available (`components/JalaliDatePicker.tsx`) for the
  Persian-calendar date inputs.

---

## Scope In

- **Date-range filter UI** on `pages/Charging.tsx` ‚Äî a "from" and "to" date control (reuse
  `JalaliDatePicker`), with a clear/reset affordance. Combine with the existing car filter
  (both applied together).
- **Wire to the API** ‚Äî extend `features/charging/api.ts` `listChargingSessions` to accept an
  optional `{ car_id?, from?, to? }` and build the query string (dates serialized as
  **RFC3339**, end-of-day handling for the inclusive "to" so a same-day range returns that
  day's sessions). Extend `useChargingSessions(filter)` to take the filter and key the query
  on it.
- **Detail view (tap-to-expand)** ‚Äî tapping a session card expands it (inline accordion or a
  dialog) to show the full record: date/time, duration, total + per-period kWh, per-period &
  total cost, location, start/end SOC (and the existing `SOCAnalysis`). Reuse the
  **TouCostBreakdown** component from TASK-0010 if available.
- **States** ‚Äî loading, empty ("no sessions in this range"), and invalid-range (from > to)
  handled gracefully; surface the API `400` as a friendly message rather than a raw error.
- **i18n + RTL** ‚Äî Persian/English labels, Jalali date display, correct under `dir="rtl"`.

## Scope Out

- Any API/DB/migration change ‚Äî the filter endpoint already exists (**no backend work**).
- New aggregate/summary numbers for the filtered range (cost trend = TASK-0011; TOU
  breakdown = TASK-0010). This task is filter + detail UI only.
- Full pagination UI / infinite scroll ‚Äî keep the existing single-page `limit=100` fetch
  unless the range is unbounded; pagination beyond 100 is a separate later task. (Note it if
  a filtered range can exceed 100.)
- Editing within the detail view ‚Äî edit already exists via the form; detail view is read-only
  (it may contain the existing Edit/Delete buttons, but no new edit flow).

---

## Acceptance Criteria

- [ ] User can pick a from/to date range; the list refetches and shows only sessions in range
      (verified against `?from`/`?to` actually sent to the API, not client-side slicing).
- [ ] Date range combines correctly with the existing car filter.
- [ ] "To" date is **inclusive** of that day (same-day from=to returns that day's sessions).
- [ ] Clearing the range restores the full list; query key changes per range (cached, no
      duplicate fetch ‚Äî ADR-002, no `useEffect` data fetching).
- [ ] Tapping a session expands a detail view showing date/time, duration, total + per-period
      kWh, per-period + total cost, location, and SOC (with `SOCAnalysis`); tapping again /
      closing collapses it.
- [ ] Empty range shows a friendly "no sessions" state; an invalid range or API `400` shows a
      friendly message, not a raw error.
- [ ] Persian + English + RTL correct; Jalali dates display correctly. `npm run build` clean.

## Open Questions ‚Äî RESOLVED (feature, 2026-06-01)

1. **Detail presentation ‚Üí DECIDED: inline accordion expand on the existing card grid.** Lowest churn ‚Äî
   the card *already* renders the full detail (TOUBreakdown, SOCAnalysis, per-period kWh) inline; we just
   gate the detailed parts behind an `expanded` flag and show a compact summary when collapsed. The grid
   row containing the expanded card grows to fit (CSS grid rows size to the tallest cell) ‚Äî acceptable. A
   modal `Dialog` is a fine alternative if the team later prefers no row-height jump, but accordion matches
   the "tap-to-expand" ask and reuses existing markup.

   date-narrowed personal EV history won't exceed 100. Pagination/infinite-scroll stays a **separate future
   task**. *Optional polish:* if `total > items.length`, show a subtle "first 100 of N ‚Äî narrow the range"
   hint (requires the hook to also expose `total`; nice-to-have, not required).

## Feature Design (feature persona ‚Äî 2026-06-01) ‚Üí hand to developer

### Grounding (verified against current code)
- **API filter exists; frontend ignores it.** `listChargingSessions()` (`features/charging/api.ts:45`) is
  hardcoded `"/v1/charging-sessions?limit=100"` (no params); `useChargingSessions()` (`hooks.ts:13`) uses a
  **static** key `["charging-sessions"]`. The Go handler already parses `?car_id&from&to` (RFC3339).
- **Car filter is currently client-side.** `pages/Charging.tsx` has `selectedCarFilter` (defaults to
  `default_car_id` once settings load) and computes `filteredSessions` by slicing client-side. A `<Select>`
  car dropdown + an Add button sit in the header.
- **The card already shows the full detail** (from TASK-0010): date, duration, total + per-period kWh,
  `<TOUBreakdown variant="inline">`, cost (with inline override edit), location, and `<SOCAnalysis>` when
  both SOCs exist. **`session.notes` is fetched but never displayed anywhere** ‚Üí good new content for detail.
- `JalaliDatePicker` (`components/JalaliDatePicker.tsx`) returns a `Date | undefined` via `onDateChange`.
- The dashboard (`Index.tsx`) calls `useChargingSessions()` **with no args** ‚Üí must stay backward-compatible.

### Decision ‚Äî filter architecture: **all filters server-side via the hook + `keepPreviousData`**
Route car + from + to through one filter object to the API and key the query on it (so the spec's "verified
against `?from`/`?to` sent to the API, not client slicing" holds, and car combines naturally). Use TanStack
`placeholderData: keepPreviousData` so changing a filter doesn't flash an empty list. Remove the client-side
`filteredSessions` car slice.

### 1) Filter bar UI (above the session list)
- **From / To** ‚Äî two `JalaliDatePicker`s (labels `ÿßÿ≤ ÿ™ÿßÿ±€åÿÆ` / `ÿ™ÿß ÿ™ÿßÿ±€åÿÆ`) + a **Clear** button (resets
  from/to; optionally car‚Üíall). State: `fromDate`, `toDate` (`Date | undefined`) in `Charging.tsx`.
- **Car** ‚Äî keep the existing `<Select>`, but **only render it when `cars.length > 1`** (a single-car user
  doesn't need it). Its value drives the filter's `car_id` (was the client slice).
- **Invalid range guard** ‚Äî if both set and `fromDate > toDate`: show a friendly inline message
  (`ÿ®ÿßÿ≤Ÿá ÿ™ÿßÿ±€åÿÆ ŸÜÿßŸÖÿπÿ™ÿ®ÿ±`) and **do not fire the query** (keeps the API from returning a raw 400).

### 2) API + hook changes (`features/charging/{api,hooks}.ts`)
- **api.ts:** add `export interface ChargingListFilter { car_id?: string; from?: Date; to?: Date }`.
  `listChargingSessions(filter?: ChargingListFilter)` builds the query with `URLSearchParams`, always
  `limit=100`, adding only set params:
  - `from` ‚Üí start of the selected day ‚Üí `.toISOString()`.
  - **`to` inclusive** ‚Üí end of the selected day (`23:59:59.999` local) ‚Üí `.toISOString()`, so `from=to`
    returns that whole day. Return type stays `ChargingSession[]` (back-compat).
- **hooks.ts:** `useChargingSessions(filter?: ChargingListFilter)` ‚Üí
  `queryKey: filter ? ["charging-sessions", filter] : ["charging-sessions"]`,
  `queryFn: () => listChargingSessions(filter)`, `placeholderData: keepPreviousData`. TanStack hashes keys
  **by value** (Dates JSON-serialize to stable ISO), so equal ranges hit the cache; the no-arg dashboard
  call keeps the base key. **Invalidation is safe:** the existing
  `invalidateQueries({ queryKey: ["charging-sessions"] })` prefix-matches the filtered keys.

### 3) Tap-to-expand detail (inline accordion)
- State `expandedId: string | null`; tapping a card's summary header toggles it (chevron indicates state).
- **Collapsed = compact summary:** car name ¬∑ date (Jalali) ¬∑ total kWh ¬∑ total cost (ÿ™ŸàŸÖÿßŸÜ). Scannable.
- **Expanded = full detail (additionally):** start time + duration ¬∑ `<TOUBreakdown variant="inline">`
  (per-period kWh + cost ‚Äî **reuse TASK-0010**) ¬∑ location ¬∑ **`notes`** (newly surfaced, when present) ¬∑
  `<SOCAnalysis>` when both SOCs exist ¬∑ the **Edit / Delete** actions (read-only detail otherwise).
- Display order: sort the rendered list **newest-first** (`started_at` desc) for a history feel (the API
  order isn't guaranteed; sort client-side for display only).

### 4) States + i18n
- **Loading** (`isLoading`) ‚Üí spinner/skeleton. **Empty-in-range** (`sessions.length === 0` with a filter
  active) ‚Üí friendly `ÿ¨ŸÑÿ≥Ÿá‚Äåÿß€å ÿØÿ± ÿß€åŸÜ ÿ®ÿßÿ≤Ÿá ŸÜ€åÿ≥ÿ™` + a Clear-filters action (distinct from the existing
  "no cars" empty state). **Invalid range** ‚Üí inline message, no fetch. **Error** (`isError`, e.g. a 400) ‚Üí
  friendly message, not a raw error.
- **i18n** (`src/i18n/locales/{en,fa}.json`) ‚Äî add: `from`, `to`, `clearFilters`, `noSessionsInRange`,
  `invalidRange`, `notes`, `details`/`showDetails`/`hideDetails`. Persian + Jalali dates + RTL throughout.

### Minor nuances (note for developer / dev_supervisor ‚Äî not blockers)
- `resetForm` prefills from `sessions[0]` (last location / start SOC); with a filter active that's the
  latest *visible* session ‚Äî acceptable.
- The "Sessions" dashboard stat card's unscoped count (carried from TASK-0011) is unrelated here.
