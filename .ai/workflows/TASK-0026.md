## Problem

TASK-0017 shipped a **combined "ورود با بله/تلگرام" tab** that picks the platform server-side
(Bale-first, Telegram fallback) based on what the user has linked. Three UX gaps remain:

1. **No platform choice on the login screen** — users who have linked both can't select; users
   who haven't linked anything see no explanation.
2. **Phone number is collected but with no explicit platform context** — the form doesn't guide
   the user to the right linked account.
3. **OTP timer / remaining-attempts feedback is absent** — wrong code gives a bare 401 with no
   count; expiry gives no countdown.

---

## Goal

Redesign the login page tabs so that **Email, Bale, and Telegram are three distinct, equally
prominent choices**. Bale and Telegram each walk the user through: pick platform → enter phone
→ receive OTP in that messenger → enter 6-digit code → logged in. Email path is unchanged.

---

## Scope In

### Login page tabs
- **Tab 1 — ایمیل (Email):** existing email + password flow, **no change**.
- **Tab 2 — بله (Bale):** phone input → OTP sent to linked Bale account → 6-digit input → login.
- **Tab 3 — تلگرام (Telegram):** phone input → OTP sent to linked Telegram account → 6-digit input → login.

### Bale / Telegram tab flow (identical structure, platform-labelled)
1. **Phone input step**
   - RTL text field, placeholder `۰۹۱۲ ۳۴۵ ۶۷۸۹` (Iranian format), accepts `0912…`, `+98912…`, `98912…`
     — server normalises to E.164 so the client doesn't need to.
   - "ارسال کد" button → calls `POST /auth/otp/request` with `{ phone, platform }`.
   - Response is always `202` (anti-enum). Show: **"کد به حساب [بله/تلگرام] شما ارسال شد."**
   - If the server returns `429`: show **"درخواست زیاد — لطفاً ۱۵ دقیقه دیگر امتحان کنید."**

2. **OTP input step** (shown after `202`)
   - Reuse existing `InputOTP` (shadcn) — 6 slots, auto-focus.
   - **60-second countdown timer**: `کد تا XX ثانیه دیگر معتبر است`. When it hits 0 → disable
     verify button, show **"کد منقضی شد — دوباره ارسال کنید"** + "ارسال مجدد" button back to step 1.
   - "تأیید" button → calls `POST /auth/otp/verify` with `{ phone, code, platform }`.

3. **Success:** access token stored in memory, silent-refresh cookie set → redirect to dashboard
   (same as email login path, ADR-003 contract).

4. **Error states on verify:**
   - Wrong code (`401`): **"کد اشتباه است — X تلاش باقی مانده"** (see backend note below).
   - Locked (`401` after exhausting attempts): **"حساب قفل شده — ۱۵ دقیقه دیگر تلاش کنید."**
   - Expired (verify called after timer ends, `401`): **"کد منقضی شد — دوباره ارسال کنید."**

5. **"شماره تلفن ثبت نشده" path:** if user enters a phone not linked to the chosen platform,
   the server still returns `202` (anti-enum) but no OTP is sent. The frontend cannot distinguish
   this from a real send. No special UI needed — after 60 s the code expires naturally. UX note:
   add static helper text below the phone field:
   > «ابتدا باید در تنظیمات حساب کاربری، شماره تلفن خود را به [بله/تلگرام] متصل کنید.»

### "ارسال مجدد" (resend) cooldown
- After first send, a **60-second resend cooldown** is enforced on the server (`otp:cooldown:<e164>` key).
- During cooldown: "ارسال مجدد" button is disabled + shows remaining seconds.
- After cooldown + before OTP expiry: "ارسال مجدد" enabled → sends a fresh code, resets the 60 s timer.

## Backend Tweaks Required (minor — in same developer pass)

TASK-0017's backend is ~95% correct. Three small tweaks are needed for this UI to work:

| # | Change | File | Detail |
|---|--------|------|--------|
| B1 | Add `platform` field to `POST /auth/otp/request` | `handler/auth.go` | Accept `"bale"` \| `"telegram"` (optional, default = Bale-first existing behaviour). Use it to select which sender to route through instead of the auto-select. Validate: if platform = telegram but no `telegram_chat_id` → still `202`, no send. |
| B2 | Add `platform` field to `POST /auth/otp/verify` | `handler/auth.go` | Namespace the Redis OTP key by platform: `otp:login:<platform>:<e164>` so Bale and Telegram OTPs don't collide for a user linked to both. |
| B3 | Return remaining attempts in `401` body | `handler/auth.go` → `service/auth.go` | On wrong-code `401`: `{ "error": "INVALID_OTP", "remaining_attempts": N }`. On lockout: `{ "error": "OTP_LOCKED" }`. |
| B4 | OTP TTL: 60 s (currently 300 s) | `service/auth.go` or constants file | Change `otp:login:…` TTL from `300s` to `60s` to match the new timer UX. |

These are single-file edits; scope them into the same developer commit as the UI work.

## Component / State Design

```
LoginPage
├── Tabs (ایمیل | بله | تلگرام)
├── EmailTab            ← untouched
├── BotLoginTab (platform="bale" | platform="telegram")
│   ├── PhoneStep
│   │   ├── PhoneInput (RTL, E.164-permissive)
│   │   └── SendButton → requestOTP(phone, platform)
│   └── OTPStep
│       ├── InputOTP (6 slots)
│       ├── CountdownTimer (60 s, disables Verify at 0)
│       ├── VerifyButton → verifyOTP(phone, code, platform)
│       ├── ResendButton (disabled during cooldown)
│       └── ErrorBanner (remaining attempts | locked | expired)
```

State machine per tab:
```
idle → sending → awaiting_code → verifying → success
                             ↗ expired (timer hits 0)
                 wrong_code ↗ (back to awaiting_code with attempt count)
                 locked      (terminal until 15 min pass)
```

All HTTP through `features/auth/api.ts` + `hooks.ts` (ADR-002). No `fetch()` in components.

## API Contracts (existing — minor extensions above)

```typescript
// features/auth/api.ts additions

export function requestOTP(phone: string, platform: "bale" | "telegram") {
  return apiPost("/auth/otp/request", { phone, platform });
}

export function verifyOTP(phone: string, code: string, platform: "bale" | "telegram") {
  return apiPost<{ access_token: string }>("/auth/otp/verify", { phone, code, platform });
}
```

Error shapes from backend (after B3):
```typescript
// 401 wrong code:   { error: "INVALID_OTP",  remaining_attempts: number }
// 401 locked:       { error: "OTP_LOCKED" }
// 401 expired:      { error: "INVALID_OTP",  remaining_attempts: number } (OTP gone from Redis)
// 429 rate-limited: standard rate-limit 429
```

## PM Decision (2026-06-05)

TASK-0017 shipped the backend correctly but the login UX combined Bale + Telegram into one opaque
server-side choice. Users need to pick their messenger explicitly so they know where to look for
the code — and the UI needs a countdown so they don't wonder if the 5-minute window expired.
This task is frontend-primary (tab split, platform labels, countdown timer, attempt counter) with
four minimal backend tweaks (platform routing, key namespace, attempt count in 401, 60 s TTL)
that are single-file edits best shipped in the same commit. Depends entirely on TASK-0017's
backend foundation (✅ CLOSED). No new migration, no new Redis infra, no new token path.
