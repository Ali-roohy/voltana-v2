## PM Decision (2026-06-05 — Revision 2)

The first pass of TASK-0026 (`f4eb0db`) shipped the three-tab login redesign and B1–B4 backend
tweaks. Two UX gaps remain:

1. **Registration has no method picker.** The "ثبت نام" tab is a plain email+password form.
   Users who want to register via Bale/Telegram must first register with email, then link the
   bot from Settings — a completely non-obvious path.

2. **No connectivity reminder.** Telegram is filtered in Iran; Bale may be down. Users need a
   visible nudge to check their connection before they wait 60 seconds for an OTP that will
   never arrive.

This revision expands TASK-0026 to add: a registration method picker, OTP-based account
creation for Bale/Telegram, and a connectivity reminder banner on all OTP steps.

---

## Problem

1. **Registration is email-only.** No path to create an account via phone + bot OTP.
2. **Missing onboarding step for OTP.** New users who've never used the bot don't know they need
   to start it and share their phone before OTP login works.
3. **No connectivity hint.** The 60-second OTP window expires silently if Telegram is filtered
   or Bale is unreachable.

---

## Goal

Unify registration and login under the same three-method frame (Email · Bale · Telegram).
Add OTP-based registration. Show a connectivity reminder the moment any OTP flow starts.

## Scope In

### 1 — Registration method picker

Replace the current plain "ثبت نام" tab with a **two-step registration flow**:

**Step 1 — Method picker** (replaces the signup form as the default view of the ثبت نام tab):
Three method cards, equally sized, each with an icon and label:

| Card | Icon | Label |
|------|------|-------|
| Email | `Mail` | ایمیل |
| Bale | `MessageCircle` | بله |
| Telegram | `Send` | تلگرام |

Selecting a card advances to Step 2 for that method.

**Step 2a — Email registration** (unchanged from current implementation):
- Full name (required)
- Email (required)
- Phone (optional)
- Password (required, min 8)
- "ثبت نام" button → `POST /auth/register`
- Back button → returns to method picker

**Step 2b — Bale registration:**
- Phone (required, E.164-permissive)
- Email (optional)
- "ارسال کد" button → shows connectivity banner → `POST /auth/otp/request { phone, platform:"bale" }`
- → 202 → OTP step (same `BotRegisterOTPStep` as login's `BotLoginTab` OTP step)
- On verify → `POST /auth/otp/register { phone, code, platform:"bale", email? }`
- Success → tokens → redirect to `/`
- Back button (phone step) → returns to method picker

**Step 2c — Telegram registration** (identical to 2b with `platform:"telegram"`).

**Prerequisite helper text** (shown below the phone input in Step 2b/2c):
> «ابتدا ربات ولتانا را در [بله/تلگرام] استارت کنید و شماره خود را با ربات به اشتراک بگذارید.»

This is a permanent hint, not conditional. It mirrors the bot onboarding instruction.

### 2 — Connectivity reminder (all OTP steps, login + registration)

Whenever the OTP request button ("ارسال کد") is tapped — before the API call fires — show a
`toast.warning` with the message:

> **«فیلترشکن رو روشن کن برای تلگرام، یا بله رو چک کن»**

This fires in both `BotLoginTab` and `BotRegisterTab`. The API call proceeds immediately after
(the toast is informational, not blocking — no confirmation required).

### 3 — Backend tweaks B5 + B6 (in same developer pass as UI)

#### B5 — Bot cold-start registration contact capture

When the Voltana bot receives `/start` **without** a link token (bare `/start`) from a user:
- Bot asks for phone with a contact-share keyboard (reuse the existing `sendContactRequest` message
  but with text: `«برای ثبت‌نام در ولتانا، شماره تلفن خود را به اشتراک بگذارید:»`)
- When the user shares their phone and there is **no pending bot-link in Redis** for that chat_id:
  - Store `reg:contact:<platform>:<phone_e164>` → `chatID` in Redis (TTL 30 min).
  - Extend `LinkCallback` interface with `StoreRegistrationContact(ctx, platform, chatID, phone)`.
  - Reply: `«✅ شماره ثبت شد. می‌توانید از اپلیکیشن ولتانا ثبت‌نام کنید.»`

If a pending bot-link exists for the chat_id, the existing linking flow runs unchanged.

Extend `RequestOTP` service: if `FindByPhone` returns not found → look up
`reg:contact:<platform>:<phone_e164>` in Redis. If found, send OTP to that chatID and store as
`otp:reg:<platform>:<phone_e164>` (same TTL / attempt logic as `otp:login:…`).

#### B6 — `POST /auth/otp/register` endpoint

New unauthenticated endpoint:

```
POST /auth/otp/register
Body: { phone, code, platform, email? }
```

- Normalise phone to E.164.
- Validate OTP from Redis key `otp:reg:<platform>:<phone_e164>` (single-use CacheGetDel,
  same attempt + lockout logic as CompleteOTPLogin).
- If phone already exists in `users` → `409 { "error": "phone already registered", "code": "PHONE_TAKEN" }`.
- If OTP valid + phone new → look up chatID from `reg:contact:<platform>:<phone_e164>` → create user
  (`email` if provided, phone, bot chat_id column based on platform) → `issueTokenPair`.
- Anti-enum: if chatID not found (user never started bot) → consume the OTP key, return `401 INVALID_OTP`
  (same as wrong code — no hint that the bot contact is missing).
- Response on success: `200 { access_token }` + httpOnly refresh cookie (same as CompleteOTPLogin).

No new migration needed: `users.bale_chat_id` / `users.telegram_chat_id` / `users.phone` columns
already exist (migration 000010).

## Feature Design (feature persona)

### Component tree

```
Auth (page)
├── [mode toggle: ورود | ثبت نام]
│
├── ورود mode
│   └── Tabs (ایمیل | بله | تلگرام)          ← already shipped in f4eb0db
│       ├── EmailLoginTab                       ← existing, unchanged
│       ├── BotLoginTab platform="bale"         ← existing + add connectivity toast
│       └── BotLoginTab platform="telegram"     ← existing + add connectivity toast
│
└── ثبت نام mode
    └── RegisterFlow
        ├── MethodPickerStep                    ← NEW: 3 method cards
        ├── EmailRegisterStep                   ← existing signup form + back button
        └── BotRegisterTab platform="bale"|"telegram"   ← NEW
            ├── PhoneStep                       ← phone + optional email + helper text
            └── OTPStep                         ← same as BotLoginTab OTP step
```

### State machines

**Auth page** (top-level):
```
mode: "login" | "register"   ← toggle button, default "login"
```

**RegisterFlow**:
```
idle                          → method picker shown
email_selected                → EmailRegisterStep shown
bale_selected                 → BotRegisterTab platform="bale"
telegram_selected             → BotRegisterTab platform="telegram"

Any "back" → idle
```

**BotRegisterTab** (same state machine as BotLoginTab):
```
phone_step
  → sending (API in flight)
  → otp_step
      → verifying
      → success
      → locked
      ← wrong_code (stay in otp_step with error)
      ← expired (timer=0)
  ← back (→ phone_step)
```

**New errors on OTP step (registration):**
```
{ type: 'phone_taken' }       → "این شماره قبلاً ثبت شده است — ورود کنید"
{ type: 'invalid'; remaining } → "کد اشتباه است — X تلاش باقی مانده"
{ type: 'locked' }            → "حساب قفل شده — ۱۵ دقیقه دیگر تلاش کنید"
```

### API additions (`features/auth/api.ts`)

```typescript
export async function registerWithOTP(
  phone: string,
  code: string,
  platform: "bale" | "telegram",
  email?: string,
): Promise<void> {
  const res = await api.post<TokenResponse>("/auth/otp/register", {
    phone, code, platform, ...(email ? { email } : {}),
  });
  authStore.setToken(res.access_token);
}
```

### Layout notes

- **Mode toggle**: Two `Button` variants (default + ghost) at the top of the Card, above the tab
  bar. Switching mode resets to the default sub-state (login → ایمیل tab; register → method picker).
- **Method picker cards**: Three equal-width cards in a `grid grid-cols-3 gap-3`. Each card:
  `Card` with `CardContent py-4 text-center cursor-pointer hover:border-primary`. Icon 24px above label.
- **BotRegisterTab**: Identical layout to `BotLoginTab`; extractable as a shared component or
  parameterised via a `mode: "login" | "register"` prop. The `verifyOTP`/`registerWithOTP` call
  switches on `mode`.
- **EmailRegisterStep**: Current signup form + a `Button variant="ghost"` "بازگشت" at the top.
