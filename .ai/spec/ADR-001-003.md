# ADR-001 — Replace Supabase with Self-Hosted Go + PostgreSQL Stack

**Date**: May 2026
**Status**: Accepted
**Decided by**: architect

---

## Context

The current MVP uses Supabase for auth, database, and RLS policies. This creates a hard dependency on a paid/external service and limits self-hosting.

## Decision

Replace Supabase with:
- **Go (Gin)** API for business logic and auth
- **PostgreSQL 16** (self-hosted in Docker) for persistence
- **Redis 7** for JWT blacklist, rate limiting, background job queue
- **golang-migrate** for versioned DB migrations

## Consequences

### Positive
- Full self-hosting — zero external dependencies
- Single Docker Compose command to run the entire stack
- Go is ideal for future OBD serial port integration
- JWT in memory (XSS-safe) instead of Supabase localStorage pattern

### Negative
- Auth must be implemented manually (register/login/refresh/logout)
- RLS policies replaced with application-layer user_id filtering (more explicit but more code)
- Email verification must be handled (SMTP setup required)

## Migration Path

1. Port schema from Supabase migrations to standard SQL files
2. Replace `auth.users` with self-managed `users` table
3. Remove all `auth.uid()` RLS references — enforce in Go repository layer
4. Replace Supabase JS SDK in frontend with plain fetch + JWT interceptor

# ADR-002 — Feature-Based Frontend Structure

**Date**: May 2026
**Status**: Accepted
**Decided by**: architect

## Context

Current frontend has Charging.tsx at 860 lines mixing data fetching, business logic, and UI. Inconsistent patterns (useEffect vs TanStack Query).

## Decision

Adopt feature-based folder structure. Each feature owns: `api.ts`, `hooks.ts`, `components/`, `Page.tsx`. Shared code only in `src/components/` and `src/lib/`.

**Rule**: No component calls `fetch()` directly. All HTTP goes through `features/<name>/api.ts` → `hooks.ts`.

## Consequences

- Charging.tsx splits into ~5 files — more readable, testable
- Consistent TanStack Query everywhere
- Feature teams (or AI agents) can work on one feature without touching others

# ADR-003 — JWT Token Storage Strategy

**Date**: May 2026
**Status**: Accepted
**Decided by**: architect + security

## Decision

- **Access token**: stored in React state (memory only). Lost on page refresh — acceptable because refresh token handles re-issue silently.
- **Refresh token**: stored in httpOnly cookie. Not accessible to JavaScript — eliminates XSS token theft.

## Rejected Alternatives

- localStorage for both — XSS-vulnerable (current Supabase approach)
- sessionStorage — still accessible to JS, lost on tab close

## Implementation

`lib/api.ts` wraps all requests. On 401 response, it calls `POST /auth/refresh` automatically (using the httpOnly cookie), receives a new access token, updates React state, and retries the original request.
